**codexclaw** (`cxc`) is a single [OpenAI Codex](https://openai.com/codex) plugin. It reuses the
Codex runtime and adds development discipline on top: a `cxc-dev` work classifier, an IPABCD
work loop driven by hooks and file state, multi-model subagent configuration over MCP, a shipped
GUI dashboard, and an optional detect-only [opencodex](https://github.com/lidge-jun/opencodex)
provider bridge.

It is not a Codex fork, not a harness, and not a provider proxy. Everything ships through the
standard Codex plugin surfaces: skills, hooks, an MCP server, local components, and a small `cxc`
CLI. The only scoped server exception is the opt-in loopback messenger bridge, which relays to
stock `codex exec` and does not orchestrate subagents.

## PABCD pipeline

<div class="cxc-rail" aria-label="codexclaw IPABCD pipeline">
  <div class="cxc-rail__step"><span class="cxc-rail__key">I</span><span class="cxc-rail__name">Interview</span><span class="cxc-rail__desc">settle ambiguity before a work loop starts</span></div>
  <div class="cxc-rail__step"><span class="cxc-rail__key">P</span><span class="cxc-rail__name">Plan</span><span class="cxc-rail__desc">write the concrete change plan and evidence target</span></div>
  <div class="cxc-rail__step"><span class="cxc-rail__key">A</span><span class="cxc-rail__name">Audit</span><span class="cxc-rail__desc">adversarial review against the real codebase</span></div>
  <div class="cxc-rail__step"><span class="cxc-rail__key">B</span><span class="cxc-rail__name">Build</span><span class="cxc-rail__desc">scoped atomic changes, verified as you go</span></div>
  <div class="cxc-rail__step"><span class="cxc-rail__key">C</span><span class="cxc-rail__name">Check</span><span class="cxc-rail__desc">real build, tests, and review gate</span></div>
  <div class="cxc-rail__step"><span class="cxc-rail__key">D</span><span class="cxc-rail__name">Done</span><span class="cxc-rail__desc">evidence summary, close to idle</span></div>
</div>

```
Codex runtime
  └─ codexclaw plugin
       ├─ skills   →  cxc-dev discipline + dev-* routers + PABCD/loop/interview
       ├─ hooks    →  PABCD state, goal guards, provider detection
       ├─ MCP      →  subagent model/prompt config
       ├─ GUI      →  dashboard + opt-in messenger bridge
       └─ cxc CLI  →  enable · status · doctor · orchestrate · gui
```

## Quick start

```bash
# Install from the public marketplace
codex plugin marketplace add lidge-jun/codexclaw
codex plugin add codexclaw@codexclaw

# Verify the install
cxc doctor

# Start a PABCD work-phase from the terminal
cxc orchestrate P --attest '{"from":"IDLE","to":"P","did":"plan the change"}'
```

The project gate is green at 801 tests. Build and test commands are documented separately so the
landing page stays a control surface, not a release note.

:::note[Status honesty]
These docs mark features `current`, `planned`, or `deferred`. A `planned` feature has a decided
design but no shipped runtime yet. The docs never describe a planned feature as if it shipped.
:::
