# codexclaw 설계철학 (CANONICAL)

Status: FOUNDING PHILOSOPHY · jun 2026-06-30 · 모든 loop 설계의 기준점(why)

> 이 문서는 codexclaw가 "왜" 이렇게 설계됐는지를 박아두는 정본이다. 어떤 loop(Ln)를
> 설계/구현할 때든, 결정이 갈리면 이 철학에 비춰 판단한다. 이건 자료글 소재가 아니라
> codexclaw의 존재 이유다.

## 문제 인식 — 결과물이 제대로 안 나오는 이유

에이전트가 "그냥 go" 했을 때 결과가 망가지는 건 세 가지가 겹치기 때문이다:
1. **의도 미전달** — 유저의 의도가 제대로 전달되지 않은 채 실행이 계속 진행된다.
2. **과거지식 고착 + 환각 빌드** — 모델이 과거 지식에 머문 상태/환각 위에서 빌드한다.
3. **편향 누적** — 한번 어긋난 방향이 교정 없이 끝까지 굴러간다.

## 원래 답 — cli-jaw의 HITL PABCD

cli-jaw는 이 문제를 **중간 검수(HITL)** 로 풀었다. IDLE에서 평범하게 대화하다가(일반 codex/claude
대화) 복잡한 작업이 오면 PABCD에 진입하고, 매 게이트에서 사람이 검증 오라클이 된다:
- **P (Plan)**: plan 에이전트가 문서화된 계획을 짠다. 안 읽는 경우가 많으니 유저에게 물어 방향을
  맞추고, **유저 승인**이 있어야 통과.
- **A (Audit)**: 직원/서브에이전트끼리 방향이 맞는지 토의. context7 + 웹서치 + 새 에이전트를
  spawn해 폴더구조만 비교해도 이상한 점이 잡힌다. 변형된 결과가 맞는지 **유저 승인**.
- **B (Build)**: TEAM 모드는 신뢰하지 않는다. A에서 확정되면 대단한 걸 만드는 게 아니므로 **solo
  빌드**. 결과 보고 후 **유저 승인**.
- **C (Check)**: 구현 후 충돌/테스트 문제가 생길 수 있으니 검증.
- **D (Done)**: 상태를 닫고 IDLE로 복귀.

## ★ 핵심 전략 — HITL과 HOTL의 병존 (전환이 아니다)

codexclaw의 전략은 HITL을 HOTL로 **갈아끼우는 것이 아니다**. 둘을 **하나의 시스템 안에 공존**시키고,
유저가 작업마다 모드를 고르게 하는 것이다. 이 병존이 곧 전략이다(자료글이 노리는 지점도 이것).

- **HITL 경로 = IDLE → IPABCD (대화형 PABCD)**: 사람이 P·A·B·C 매 게이트의 검증 오라클로 남는다.
  고위험·모호·새 도메인일 때 이 경로를 쓴다. 매 단계 유저 승인으로 통과.
- **HOTL 경로 = /goal (자율 PABCD)**: Interview로 정렬이 끝났을 때, 사람이 루프 위(on the loop)로
  물러나고 PABCD가 자율 실행된다. 흐름: 아이디어 공유 → P → A → B → C 자율검증 → D 종료.
- 두 경로는 **같은 PABCD 단계를 공유**하되 **구동 모드가 다르다**. loop engineering(모델 발전으로
  P를 에이전트가 알아서 세울 수 있게 된 것)은 HOTL 경로를 *가능하게* 만든 것이지, HITL을 *없앤*
  것이 아니다.

## 그래서 FSM 분리가 codexclaw의 핵심 요소다

병존이 성립하려면 **FSM이 세 가지를 깨끗하게 분리**해야 한다. 이게 codexclaw 설계의 중심축이다:
1. **FSM phase 로직** — I/P/A/B/C/D가 지금 어디인지(단계 오케스트레이션). 구동 모드와 직교.
2. **IDLE ↔ IPABCD (HITL 모드)** — 게이트마다 사람 승인을 요구. Interview/`request_user_input`
   사용 가능.
3. **goal (HOTL 모드)** — 자율 실행. 사람 게이트 없음. 그래서 **goal 모드에서는 Interview/
   `request_user_input`을 hard-deny**(L11)해야 모드 경계가 무너지지 않는다.

즉 L11의 "goal 모드 인터뷰 금지"는 부가 규칙이 아니라 **HITL/HOTL 병존을 성립시키는 경계 조건**이다.
모드가 섞이면(자율 실행 중 사람에게 묻거나, 대화형인데 승인 없이 질주) 두 전략이 동시에 깨진다.

## 받침대 — dev skill + 검색 가이드

병존만으로는 부족하다. **dev skill(모듈형, LOC 큼) + 웹서치 가이드 주입**으로 자료조사·구현 결정이
어긋나지 않게 받친다. 특히 HOTL 경로에서 이 안전벨트가 자율성의 품질을 직접 결정한다. 그 결과 요즘은
아키텍처상의 오류는 거의 발생하지 않는다.

## 단일 실패점 — HOTL 경로에서 Interview 하드닝이 핵심

HOTL 경로(/goal)에서는 사람 게이트가 맨 앞 정렬 하나뿐이라 **Interview 품질이 단일 실패점**이 된다.
의도가 어긋난 채 /goal을 돌리면 자율 PABCD가 "틀린 것을 아주 성실하게, 높은 확신으로" 빌드한다.
(HITL 경로는 게이트가 여러 개라 이 위험이 작다 — 그래서 모호하면 IPABCD로 가는 선택지가 존재한다.)
이걸 막는 장치가 L8~L11이다:
- 모순탐지 서브에이전트(metis / 5-Mind)를 Interview 안으로 끌고 들어와 negativity-bias 재질문
  루프를 돌린다 → 단일 게이트를 단단하게.
- fresh-eyes 리셋 프롬프트(M3) + 독립 리뷰어로 편향 누적을 끊는다.

## 두 종류의 drift, 두 종류의 게이트

"그냥 go 했을 때 계속 가는 문제"는 두 축이다:
- **방향 drift (엉뚱한 걸 함)** → **Interview**가 맨 앞에서 잡는다.
- **스코프 drift (맞는 걸 하다가 안 멈추고 번짐)** → **goal/attest 게이트 + D의 IDLE 복귀**가 잡는다.

HOTL이 게이트를 *두 종류* 갖는 이유가 이것이다. Interview만으로는 스코프 drift를 못 막는다.

## 3대 실패모드 → 처방 (1:1)

| 실패모드 | 처방 |
|----------|------|
| 과거지식 고착 | websearch / context7 주입 |
| 환각 빌드 | A audit (새 에이전트 spawn → 폴더구조 비교 + 토의) |
| 편향 누적 | fresh-eyes 리셋 프롬프트(M3) + 독립 리뷰어 |

## 설계 결정에 주는 함의 (loop 짤 때 적용)

- HITL/HOTL **병존이 전략**이다. 둘 중 하나로 수렴시키지 않는다. 유저가 작업마다 모드를 고른다.
- 그 병존을 성립시키는 것이 **FSM 분리**(phase / IDLE-IPABCD / goal)다 → 모드 경계를 코드로 지킨다.
- L11 "goal 모드 인터뷰 금지"는 경계 조건이다. 약하게 만들면 두 모드가 섞여 둘 다 깨진다.
- HOTL 경로에서 Interview는 단일 실패점이다 → L8~L11 우선, 약하게 만들면 자율 경로가 무너진다.
- 자율성은 "사람을 뺀 것"이 아니라 "사람을 루프 위로 옮긴 것"이다 → /goal 진입 전 정렬, 진입 후 비개입.
- dev skill + 검색 가이드는 HOTL의 안전벨트다. 스킬 실콘텐츠 포팅(L12~L19)이 자율성 품질을 결정한다.
- 게이트는 방향용(Interview)과 스코프용(goal/attest·D)을 분리해서 유지한다.
