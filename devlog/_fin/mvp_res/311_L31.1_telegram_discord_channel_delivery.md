# L31.1 (Decade 311) -- Telegram and Discord Channel Delivery

Status: PLANNED defer
Cluster: 6 - Phase: expansion - Shorthand: cxc
Source-of-record: 310_L31_channel_delivery.md, 260629_codexclaw_mvp/150_channel_delivery.md, 090 J-2

## Goal (one slice)
Add deferred telegram/discord forwarding for scheduled run results after L30
defines the primary result store.

## Why now / dependencies
- Upstream: depends on L30 result contract; channel delivery forwards stored
  results and never owns the primary copy.
- Downstream: enables heartbeat-style notifications outside the terminal while
  keeping Phase 3 shippable without channel credentials.

## Scope (decision-complete)
- Files to add/edit:
  - channel delivery module or CLI under `cxc`
  - channel config store following existing codexclaw config conventions
  - telegram and discord adapter tests with fake transports
  - schedule notification opt-in wiring after L30
- Exact behavior:
  - Configure a named telegram or discord target.
  - Test delivery with `cxc channel test <name>`.
  - Opt a scheduled job into forwarding an existing L30 run summary.
  - Missing credentials or delivery failures produce explicit errors.
- Must-NOT-Have:
  - No channel credentials in docs, logs, fixtures, or transcripts.
  - No channel delivery as the only result copy.
  - No forced setup during MVP install.
  - No telegram/discord coupling inside L29 or L30.

## IPABCD micro-cycle
- I: not interview-bearing unless credential prompts are explicitly added later.
- P: define config shape, adapter interface, redaction rules, test transports,
  and schedule opt-in command.
- A: audit angle = "can notification failure hide or corrupt the stored run?"
  Reviewer checks secret handling and failure isolation.
- B: implement configure/test/send primitives, fake transports, and schedule
  notify opt-in.
- C: run adapter tests; run `cxc channel test` in dry-run or mock mode; verify a
  failed send leaves L30 result unchanged.
- D: done = telegram/discord forwarding is optional, testable, and isolated.

## Acceptance (1-3 testable criteria)
1. `cxc channel test <name>` works in fake/mock tests with clear stdout.
2. A scheduled job can forward an existing L30 run summary without modifying the
   stored result.
3. Missing credentials and delivery failures never delete or mask run results.

## QA channel (node:test path / CLI stdout / tmux / data dump)
- node:test with fake telegram/discord transports and redaction checks.
- CLI stdout for configure/test dry-run paths.
- Data dump comparing pre/post L30 run result after failed delivery.

## Commit unit (one atomic conventional commit)
`feat(channels): add optional telegram and discord forwarding`

## Blocked-on (jun decision id, if any)
None. This loop is planned and deferred; it starts only after L30 settles
Q-P3-2 because channels forward scheduled results rather than defining them.

Deferred options: CLI-only setup (recommended), skill-driven senders, or GUI
settings. Impact: CLI-only keeps the first deferred pass simple and scriptable;
later GUI can reuse the same adapter primitives.
