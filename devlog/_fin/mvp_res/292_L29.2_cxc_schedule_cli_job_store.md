# L29.2 (Decade 292) -- cxc schedule CLI and Job Store

Status: DEFERRED (Q-P3-1, Phase 3)
Cluster: 5 - Phase: 3 - Shorthand: cxc
Source-of-record: 290_L29_scheduler_mechanism_cli.md, 260629_codexclaw_mvp/040_phase3_overview.md

## Goal (one slice)
Add `cxc schedule` commands and a project-local `.codexclaw/schedule.json` job
store for recurring `codex exec` work.

## Why now / dependencies
- Upstream: depends on L29.1 mechanism boundary and L20 only for CLI naming
  conventions.
- Downstream: L30 reads job/run metadata from this store and verifies Phase 3.

## Scope (decision-complete except Q-P3-1)
- Files to add/edit:
  - CLI schedule command under existing `cxc` surface
  - `.codexclaw/schedule.json` schema/store module
  - tests for add/list/remove and interval parsing
- Exact behavior:
  - `cxc schedule add --every 15m --prompt "..."` creates a stable job record.
  - Store records include job id, enabled flag, schedule object, prompt, command
    metadata, and timestamps.
  - Include `cxc schedule list`, `cxc schedule remove <id>`, and dry-run.
  - Dry-run prints generated scheduler artifacts and does not write system state.
- Must-NOT-Have:
  - No global daemon database.
  - No channel delivery logic.
  - No silent acceptance of malformed intervals or empty prompts.

## IPABCD micro-cycle
- I: not interview-bearing; Q-P3-1 controls whether add installs OS artifacts
  or only stores/previews them.
- P: define schema, interval parser, command handlers, dry-run mode, and
  remove/list output.
- A: audit angle = "can users understand and reverse what schedule add did?"
  Reviewer checks store record, generated artifacts, and remove behavior.
- B: implement store read/write, CLI handlers, interval validation, and writer
  integration.
- C: run node:test; run `cxc schedule add --every 15m --prompt "..." --dry-run`,
  `list`, and `remove` stdout captures.
- D: done = a project can record, list, and remove recurring jobs through cxc.

## Acceptance (1-3 testable criteria)
1. Dry-run add prints job id, store preview, and scheduler artifact.
2. `.codexclaw/schedule.json` roundtrips enabled jobs and rejects bad input.
3. Remove deletes the job record and reports the scheduler removal action.

## QA channel (node:test path / CLI stdout / tmux / data dump)
- node:test for job-store schema, interval parsing, and CLI handlers.
- CLI stdout: add dry-run, list, and remove.
- Data dump: `.codexclaw/schedule.json` fixture before and after remove.
