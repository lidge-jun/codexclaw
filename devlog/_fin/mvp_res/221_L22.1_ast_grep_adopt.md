## Goal (one slice)
Adopt omo `ast-grep` as the only MVP code-intelligence addition: structural
search and codemod support backed by lazy `sg` provisioning. This is the resolved
130-defer Option A path: ast-grep is in, while LSP/codegraph are out of MVP.

## Why now / dependencies
- Upstream: L12-L17 dev skill ports define when structural search should be
  recommended instead of plain `rg`.
- J-13 resolved install policy: follow lazycodex-style provisioning rather than
  PATH-only or default exclusion.
- J-16 keeps heavy daemon/indexing systems out of core; ast-grep is acceptable
  because it is a skill plus helper plus external binary, not a persistent MCP
  service.
- Downstream: L19 skill-hub catalogs ast-grep as an on-demand code-intelligence
  skill, and `cxc doctor` can report binary/provisioning health.

## Scope (decision-complete)
Files to add/edit in implementation:
- `plugins/codexclaw/skills/ast-grep/SKILL.md`
- `plugins/codexclaw/skills/ast-grep/scripts/ast_grep_helper.py`
- `plugins/codexclaw/skills/ast-grep/references/install.md`
- `plugins/codexclaw/skills/ast-grep/agents/openai.yaml`
- `plugins/codexclaw/src/cli/doctor.*` for `sg` provisioning/reporting

Exact behavior:
- ast-grep is for AST-aware structural search, deterministic codemods, and YAML
  rule scan/apply across supported languages.
- Plain filename, literal text, regex, and simple candidate callsite search still
  use `rg` first; the skill must say not to use ast-grep for ordinary grep work.
- The omo helper is absorbed with Codex/codexclaw path substitutions and no omo
  runtime path as the primary location.
- Binary resolution checks codexclaw-managed runtime first, then explicit env/path
  locations, then PATH/Homebrew-style fallbacks if documented.
- Lazy provisioning ensures `sg` when the skill or doctor path needs it. It should
  be evidence-bound, idempotent, and safe to fail with a clear install hint.
- `cxc doctor` reports ast-grep status: provisioned, PATH fallback, missing, or
  provisioning failed, with binary path and version when available.
- `cxc` exposure is not a broad CLI replacement for the skill. The primary user
  surface is the skill; `cxc doctor`/possibly `cxc ast-grep doctor` only reports
  or prepares runtime health if implementation chooses a subcommand.

Must-NOT-Have:
- No LSP, codegraph, daemon, socket, background indexer, or MCP server in this
  sub-pass.
- No mandatory network download during prompt rendering; provisioning happens
  only when invoked by skill helper, doctor, or explicit setup path.
- No silent fallback from failed structural rewrite to unsafe regex replacement.
- No vendored binary without documented provenance and update path.
