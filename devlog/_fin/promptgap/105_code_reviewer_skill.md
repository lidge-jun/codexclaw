# 105 — Code Reviewer Skill Prompt Gaps

Gap class: PROMPT/SKILL · evidence: explorer #105

> codexclaw's `dev-code-reviewer` has the right topic coverage, but the review contract is still softer than the omo/lazycodex reviewer stack and the jaw reviewer prompt. The main misses are discipline misses: no manual-findings ordering contract (tool-findings-first already exists at `:62`), weaker proof requirements, softer blocker semantics, and almost no adversarial "trust nothing" framing.

## Reinforcement shape

1. Add a short `Review posture` preamble near the top: skeptical, independent, executor claims untrusted, tests untrusted until inspected, regression hunter first.
2. Add a mandatory `Review output contract` section with exact order: `tool findings` -> `manual findings sorted by severity` -> `blocking_issues` -> `verdict` -> `confidence`.
3. Promote proof requirements from style advice to gate language: every finding needs `trigger`, `impact`, `path:line`, and patch/new-behavior anchor.
4. Replace the current verdict table with deterministic blocker semantics: unresolved Critical/High cannot approve.
5. Add a dedicated `Regression and false-confidence tests` subsection that explicitly hunts deletion-only tests, tautological tests, implementation-mirroring tests, and scope-drift abstractions.
6. Add a `Cross-boundary proof` subsection so reviewers must inspect the consumer-side dispatch path, not just the emitting diff hunk.
7. Keep the existing antipattern catalog, but demote it behind the audit contract so the skill reads as a review discipline first and a smell encyclopedia second.
