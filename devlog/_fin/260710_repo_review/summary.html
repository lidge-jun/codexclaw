PABCD가 뭔가요?

P
Plan
→
A
Audit
→
B
Build
→
C
Check
→
D
Done
Plan: 무엇을 할지 구체적인 계획을 세운다. 파일 경로, 범위, 검증 방법까지 사전에 확정한다.
Audit: 독립 리뷰어(다른 AI 에이전트)가 계획을 적대적으로 검증한다. PASS를 받을 때까지 반복한다.
Build: 계획대로 실제 작업을 수행한다. 이번엔 검토 문서를 작성했다.
Check: 결과물이 수용 기준을 만족하는지 기계적으로 검증한다. 인용 개수, 원본 대조 등.
Done: 증거와 함께 사이클을 마감한다. 목표가 남아 있으면 다음 사이클의 P로 돌아간다.
이 루프를 4번 돌렸다. 매 사이클마다 독립 리뷰어 에이전트가 계획을 까다롭게 검증했고, 사이클 1에서만 8라운드를 거쳤다. 총 12라운드의 감사를 통과한 뒤에야 산출물이 나왔다.

교차 발견: 같은 원인, 여러 표면

1. Fail-open 완료 우회

goalplan 파일이 없거나 깨지면 readGoalplan()이 null을 반환하고, 완료 guard의 if (plan) 밖으로 빠져 E8 검증이 통째로 건너뛴다. evidence도 문자열이 비어 있지 않은지만 확인하기 때문에 아무 글자나 넣으면 통과한다. 문서는 이보다 훨씬 강한 보장을 주장하고 있었다.

011#H-1 · 031#H-1 · 031#H-2 · 021#H-1

2. 문서가 말하는 것 vs 코드가 하는 것

주석은 E8을 "final D-close gate"라고 부르지만 실제 D-close 경로에는 E8 호출이 없다. structure 문서는 V2가 기본이라 하고 pabcd 스킬은 V1이 기본이라 한다. reasoning effort 설정은 저장되지만 실제 spawn payload에 반영되지 않는 dead config다.

011#M-3 · 011#M-5 · 021#H-3 · 031#M-1

3. 수동 관리되는 진실의 원천(SoT)

CLI 명령 목록이 세 군데에 중복되고, skill visibility 집합이 canonical과 INDEX에서 다르고, deprecated 스킬이 소유자 표에 현행처럼 남아 있다. 선언을 여러 문서에 손으로 복사하는 구조가 스테일과 의미 확장을 만든다.

011#L-3 · 021#M-1 · 021#M-2 · 021#M-3 · 031#L-1

4. 상태 경계와 원장 무결성

render ledger가 session별로 격리되지 않고, plateau 판정이 work-phase 경계를 무시하고, state 저장과 ledger append가 비원자적이다. goalplan ledger도 절반 가까운 행이 선언 타입과 다른 필드를 쓴다. interview ledger에는 동일 eventId 중복이 남아 있다.

주요 발견 사항

전체 33건 중 High 5건, Med 19건, Low 9건. 아래는 가장 중요한 것들만 추렸다.

High 손상된 goalplan이 완료 gate를 우회
goal-gate.ts:190의 if (plan) 때문에 goalplan이 없거나 파싱에 실패하면 E8 검증 없이 goal 완료가 허용된다. 파일 부재와 파싱 오류를 모두 null로 접는 reader와 결합해 사실상 goalplan을 지우기만 하면 아무 goal이든 닫을 수 있다.
High spawn guard 토큰이 child에 노출, 자가 우회 가능
leaf guard의 지시문이 CXC-SUBSPAWN-ALLOWED 토큰 이름을 child에게 직접 보여준다. deny 조건이 단순 includes() 체크라서 child가 토큰을 자기 메시지에 넣으면 재귀 spawn 제한뿐 아니라 전체 leaf guard가 해제된다.
High D-close가 미완료 task를 증빙 없이 완료 처리
advanceWorkPhase()가 현재 phase의 모든 task를 무조건 done으로 바꾼다. E8은 task를 순회하지 않으므로 실제로 하지 않은 작업도 완료된 것으로 기록된다.
High A-gate enforcement가 문서와 런타임에서 정반대
structure 문서는 reviewer dispatch를 "구조적으로 보장한다"고 쓰지만, pabcd 스킬은 "gate가 출처를 검증할 수 없는 form-only bar"라고 명시한다. 또 다른 structure 문서는 self-audit도 허용해 mandatory reviewer 계약을 약화한다.
Med reasoning effort 설정이 dead config
RoleConfig.effort와 resolver가 존재하지만 V2 builder와 hook 모두 payload에 넣지 않는다. 설정해도 아무 효과가 없는 유령 설정이다.
Med V1/V2 기본값이 문서마다 반대
structure/10은 V2가 live surface라고 쓰고, pabcd/SKILL.md와 structure/60은 V1 default/V2 opt-in이라 한다. 실행 시 잘못된 namespace와 lifecycle verb를 선택하게 될 수 있다.
Med FSM state와 ledger 기록이 비원자적
state를 먼저 저장한 뒤 ledger를 append하는데, 후자가 실패하면 CLI의 fail-open catch가 오류를 삼켜서 phase만 전진하고 감사 기록은 빠질 수 있다.
Low CLI 명령 목록이 세 군데에서 수동 중복
파일 헤더 주석, TOP_LEVEL_HELP 문자열, switch/case 블록이 따로 관리된다. 명령을 추가할 때 셋 중 하나를 빼먹으면 도움말과 실제 표면이 어긋난다.

권고 Top 5

1

goalplan 파싱 실패를 fail-closed로 전환

session에 slug가 있는데 plan이 없거나 깨졌으면 완료를 거부한다. evidence도 경로/exit code/시각을 구조적으로 검사한다.

2

D-close와 완료 의미를 분리

D-close는 사이클 종료만 기록하고, task/criterion 완료는 개별 증거가 있을 때만 전이시킨다. 자동 done 처리를 제거해 원장의 진실성을 보존한다.

3

CLI/skill/structure 표를 단일 schema에서 생성

명령 목록, visibility 집합, deprecated map을 기계 판독 가능한 한 곳에 두고 help/INDEX/structure를 자동 생성한다. CI에서 drift를 잡는다.

4

state와 ledger를 versioned schema로 통합 검증

session/workPhaseId를 필수 key로 만들고, event union/중복 ID/교차 필드 불변식을 validator에서 검사한다. 과거 loose row는 version별 decoder로 읽는다.

5

subagent lifecycle/token/effort 계약을 payload 테스트로 고정

V1/V2별 canonical payload fixture에서 TASK packet, recursion grant, model/effort, leaf guard를 end-to-end assertion한다.
