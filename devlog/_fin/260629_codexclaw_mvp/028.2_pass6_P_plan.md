## Ground truth (source-verified this phase; A-audit Lorentz corrections folded)
- `codex features enable <name>` / `disable <name>` is a real persistent subcommand
  (`codex-rs/cli/src/main.rs:1436/1444`) — it writes `features.<name>=true/false` into config.toml using
  format-preserving `toml_edit` (`core/src/config/edit.rs:875,1194`), so it does NOT reorder/normalize the
  rest of the file. (The global `--enable/--disable` flags at main.rs:749-754 are ephemeral `-c` overrides.)
- Declared flags (022.2) with VERIFIED stages (features/src/lib.rs):
  - `multi_agent` — Stable, default true.
  - `goals` — Stable, default true.
  - `hooks` — Stable, default true.
  - `default_mode_request_user_input` — **UnderDevelopment, default FALSE** (lib.rs:1089-1092). Enabling it
    emits an "under-development features enabled" stderr warning (main.rs:1683). This is the one flag that is
    actually OFF by default and that codexclaw must turn on for the Interview phase.
- `multi_agent_v2` — UnderDevelopment, default false (lib.rs:935): NOT depended on, never flipped.
- codex config writes are atomic tempfile+persist with NO `.bak` (utils/path-utils). codexclaw's own backup
  is a NEW safeguard, not a mirror of codex behavior.
- Component convention (pabcd-state): Node v24 native TS, ESM, `node:fs`, `node --test`, no build step.

## Design decision (pivot per audit): wrap the official CLI, do NOT hand-roll a TOML editor
codexclaw activation SHELLS OUT to `codex features enable <name>` for each declared flag that is not already
enabled. Rationale: the official path is already format-preserving (toml_edit) and is the sanctioned,
user-aware mechanism (027). A hand-rolled line editor would re-implement toml_edit's edge cases (dotted
`features.x=true`, inline tables, quoted keys, trailing comments, commented-out keys, duplicate/profile
`[features]` tables, BOM/no-final-newline) for no benefit and more risk. codexclaw's value-add is the
pre/post snapshot + revert manifest + backup around that official call.

## Scope boundary
- IN: new component `plugins/codexclaw/components/config-guard/`:
  - `src/features.ts` — `DECLARED_FEATURES`, pure helpers: read the effective state of the 4 flags by
    invoking `codex features list` (parse names→enabled) given an injected runner; compute which flags need
    enabling (only those not already true). NO direct TOML parsing/editing.
  - `src/activate.ts` — orchestration with INJECTED deps (a `runCodex(args)` runner + an explicit
    `codexHome`/`configPath`): snapshot config.toml bytes + `codex features list` before; copy a timestamped
    `.bak`; for each not-yet-enabled declared flag run `codex features enable <name>`; record an activation
    manifest (`<codexHome>/.codexclaw-install.json`: per flag {priorEnabled, addedByCodexclaw}) for exact revert.
  - `src/deactivate.ts` — read the manifest; for each flag codexclaw enabled whose prior state was false/absent,
    run `codex features disable <name>` (or remove). Pre-existing-true flags are left untouched. Drift guard:
    if the live config no longer matches the post-activate snapshot hash, do NOT blindly revert — report and
    no-op the drifted keys (safe-by-default).
  - `src/cli.ts` — `enable`/`disable`/`status` subcommands; resolves the real codex runner + codexHome only
    HERE (the lib layer never defaults to homedir()).
  - `test/features.test.ts` + `test/activate.test.ts` — use a FAKE `runCodex` (records calls, simulates
    `features list` output + config mutation) and a `mkdtemp` temp dir. Assert: only declared flags enabled,
    only not-already-true ones get an enable call, manifest correctness, revert restores prior state, idempotent
    re-enable. A guard throws if any resolved path equals the real `~/.codex/config.toml`.
  - `package.json` mirroring pabcd-state (incl. the harmless `"main":"dist/cli.js"`).
- OUT: running against the user's real `~/.codex/config.toml` (tests are fully offline w/ fake runner + temp dir);
  provider/model keys (ocx, Phase 2); marketplace publishing (Pass 7); GUI.

## Key invariants
- codexclaw enables ONLY the 4 declared flags, and only those not already enabled (no redundant writes).
- Format/byte preservation of unrelated config is delegated to the official `codex features` (toml_edit) — not
  re-implemented. codexclaw still snapshots + backs up so a revert is always possible.
- enable→disable restores each flag codexclaw turned on to its prior state; pre-existing-true flags untouched.
- `default_mode_request_user_input` is soft (Q-IPABCD-2): if enabling fails or the flag is unavailable in this
  build, log it and continue — Interview degrades to plain-text Q&A rather than hard-failing activation.
