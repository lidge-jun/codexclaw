# L21.2 (Decade 212) -- omo Role Absorb

Status: ANALYZED
Cluster: 3 · Phase: expansion · Shorthand: cxc
Source-of-record: 210_L21_subagent_roles_ops.md; 260629_codexclaw_mvp/140_subagent_roles_ops.md

## Goal (one slice)
Map omo's ten role files into codexclaw's three inline role templates as variants,
not new first-class MVP roles. This preserves useful instructions while avoiding
role-registration and prompt-size complexity.

## Why now / dependencies
- Upstream: L21.1 defines the three role containers.
- J-8 says `.toml` can be source-of-truth but MVP injects role instructions inline.
- J-14 says ops roles and `mind-*` roles share a taxonomy, so omo variants should
  not create an uncontrolled namespace.
- Downstream: L24/L27 can later expose variants in GUI/model settings.

## Scope (decision-complete)
Files to add/edit in implementation:
- `plugins/codexclaw/roles/variants/*` or equivalent template sections
- `plugins/codexclaw/docs/roles.md` or role mapping reference
- Role template tests

Exact behavior:
- omo `explorer` maps to codexclaw `explorer`: local read-only codebase search,
  absolute paths, evidence-first reporting.
- omo `librarian` maps to `explorer` external variant: OSS/docs research,
  source URLs, permalinks where relevant, wired to L18 search hub.
- omo `plan` maps to the PABCD Plan helper, not a default subagent role.
- omo `metis` maps to `reviewer` plan-risk variant: gaps, ambiguity, constraints,
  and pre-planning risks.
- omo `momus` maps to `reviewer` plan-gate variant: OKAY/ITERATE/REJECT style
  executability review.
- omo `lazycodex-code-reviewer`, `lazycodex-gate-reviewer`, and
  `lazycodex-clone-fidelity-reviewer` map to `reviewer` variants for code,
  final gate, and frontend/design fidelity.
- omo `lazycodex-executor` maps to `executor` default implementation variant.
- omo `lazycodex-qa-executor` maps to `executor` QA variant: real-surface QA,
  evidence matrix, no product edits unless explicitly assigned.

Must-NOT-Have:
- No ten new default roles.
- No model/service-tier guarantees from omo `.toml` in MVP inline calls.
- No copied tool names that do not exist in Codex.
- No `.omo/plans` state path; codexclaw uses its own state conventions.

## IPABCD micro-cycle
- I: not interview-bearing.
- P: create a mapping table and variant snippets under the three role families.
- A: reviewer checks every omo role is either absorbed, deliberately deferred, or
  mapped to PABCD helper.
- B: port useful developer instructions with Codex tool names and codexclaw paths.
- C: static test ensures all ten source role names appear exactly in mapping docs.
- D: done = role behavior is preserved without expanding MVP role count.

## Acceptance (1-3 testable criteria)
1. Mapping docs account for all ten omo roles.
2. Runtime role registry still exposes only explorer/reviewer/executor for MVP.
3. Variant snippets avoid non-Codex tool names and `.omo` state paths.

- Role docs snapshot for the mapping table.
