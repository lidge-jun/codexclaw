# L20.4 (Decade 204) -- cxc chat-search Wrapper

Status: RETIRED (superseded by mvp_hard L13/WP1, D1' 2026-06-30)
Cluster: 3 · Phase: expansion · Shorthand: cxc
Source-of-record: 200_L20_clijaw_command_mapping.md; 260629_codexclaw_mvp/090_clijaw_command_mapping.md; 090.1 J-7

> RETIREMENT NOTE (2026-06-30, interview-confirmed D1'): `cxc chat-search` has been
> removed from the codebase. The wrapper existed because Codex app-server `thread/search`
> has no native CLI/agent surface — but on review that absence makes the wrapper a
> codexclaw-invented search surface, which conflicts with the L10/L20.4 boundary
> ("no self-implemented search; route through native"). Native `thread/search` exposing
> no CLI/agent surface is now recorded as an explicit NON-GOAL. Repository/log/config
> and external lookups route through the `cxc-search` skill (the native search ladder).
> The section below is preserved for history only; the command no longer ships.

## Goal (one slice)
Add `cxc chat-search` as a thin wrapper over Codex app-server `thread/search` so
codexclaw can recover prior thread snippets without recreating cli-jaw memory or
chat database infrastructure.

## Why now / dependencies
- Upstream: L6 provides the `cxc` CLI; L20 owns the command mapping.
- J-7 keeps cli-jaw memory as PASS to Codex built-in memory, so chat search must
  stay a wrapper over existing Codex protocol rather than a new memory system.
- Downstream: supports recall workflows and diagnostics without coupling to a
  jaw server or dashboard.

## Scope (decision-complete)
Files to add/edit in implementation:
- `plugins/codexclaw/src/cli/chat-search.*`
- `plugins/codexclaw/bin/codexclaw.mjs` or command registry equivalent
- `plugins/codexclaw/test/*chat-search*`
- Command mapping reference docs

Exact behavior:
- Command form: `cxc chat-search "<query>" [--limit N]`.
- The wrapper connects to Codex app-server when available and calls the
  `thread/search` protocol with `search_term` set from the query.
- Output includes thread identifier or title when available, snippet, timestamp
  or ordering metadata when available, and a clear no-results message.
- If app-server is not running or protocol support is unavailable, fail clearly
  with the checked endpoint/protocol and next action. Do not silently switch to
  filesystem grep or jaw chat search.
- The command is read-only.
- Limit defaults should be small and documented; input query is required.

Must-NOT-Have:
- No local chat indexer, memory store, or dashboard dependency.
- No fallback to `cli-jaw chat search`.
- No hidden startup of a durable app-server unless the Codex runtime explicitly
  supports and documents that path.
- No exposure of secrets or raw internal protocol tokens in error output.

## IPABCD micro-cycle
- I: not interview-bearing.
- P: add CLI command parser and protocol client wrapper with explicit unsupported
  errors.
- A: reviewer checks this remains a wrapper, not a second memory implementation.
- B: implement query validation, app-server call, result formatting, and failure
  messages.
- C: run node:test against a mocked `thread/search` response and a failure case;
  capture CLI stdout.
- D: done = `cxc chat-search` returns snippets or a precise unsupported/no-server
  message.

## Acceptance (1-3 testable criteria)
1. Missing query exits nonzero with usage text.
2. Mocked `thread/search` results render as thread/snippet rows.
3. App-server unavailable failure is explicit and does not try another backend.

## QA channel (node:test path / CLI stdout / tmux / data dump)
- node:test with mocked app-server protocol responses.
- CLI stdout fixtures for success, no results, and unavailable protocol.

## Commit unit (one atomic conventional commit)
`feat(cli): add cxc chat-search app-server wrapper`

## Blocked-on (jun decision id, if any)
None. If app-server protocol shape changes, implementation should fail with an
unsupported message rather than invent a fallback.
