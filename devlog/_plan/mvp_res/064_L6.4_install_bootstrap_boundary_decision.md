# L6.4 (Decade 064 slot) -- R-13 decision: install bootstrap boundary (codex owns hook-trust + plugin enable)

Status: DONE (007 R-13, decision-record)
Cluster: 1 (L6 hardening) - Phase: 1 - Shorthand: cxc
Source-of-record: 000_research_src/007_impl_reality_findings.md (R-13); codex-rs (121_openai-codex)
core-plugins/src/manager.rs, config/src/plugin_edit.rs, tui/src/startup_hooks_review.rs, tui/src/hooks_rpc.rs

## Decision
codexclaw does NOT hand-write `config.toml`, does NOT re-stamp hook trust, and does NOT add an
omo-style bootstrap step. The "config untouched except `codex features enable`" invariant STANDS.
Bootstrap responsibilities omo performs manually are owned by the codex runtime/installer here.

## Why (codex-rs measured evidence, 121_openai-codex)
1. **Plugin enable is codex's job, written by codex.** `config/src/plugin_edit.rs:21`
   `set_user_plugin_enabled(...)` writes the plugin entry into config.toml (tests:
   `set_user_plugin_enabled_writes_plugin_entry`, `..._preserves_existing_plugin_fields`,
   `..._follows_config_symlink`). `core-plugins/src/manager.rs` consumes
   `codex_config::set_user_plugin_enabled` + `codex_hooks::plugin_hook_declarations`. So a plugin's
   hooks are declared/loaded by the manager from the plugin manifest — codexclaw does not register
   them by editing config.toml.
2. **Hook-trust is a codex startup gate, not a plugin responsibility.**
   `tui/src/startup_hooks_review.rs:45` `maybe_run_startup_hooks_review(...)` presents
   `TrustAllAndContinue` / `ContinueWithoutTrusting`; trust is persisted via
   `hooks_rpc::write_hook_trusts` and gated by `hook_needs_review`. codex re-stamps/reviews hook
   trust itself; a plugin must NOT forge trust entries (omo's manual re-stamp is an omo-runtime
   concern, not a codex-plugin requirement).
3. **Admission trust model.** `manager.rs:~429` documents that after marketplace admission for a
   CODEX_HOME, runtime plugin loading TRUSTS that CODEX_HOME's contents. The trust boundary is
   admission-time (install), enforced by codex — not something the plugin bootstraps.

## Consequence for L6 (config-guard)
- `config-guard` correctly limits itself to `codex features enable` (soft-fail tolerant) + an install
  manifest for revert. It must NOT grow a config.toml writer or a hook-trust stamper.
- If `codex features enable` is unavailable in a given codex build (soft feature), the existing
  soft-fail path (activate.ts SOFT_FEATURES / enableFailed) is the correct degraded behavior — NOT a
  hand-written TOML fallback.

## Revisit triggers
- If a future codex build stops auto-loading plugin-declared hooks (requires explicit per-hook config
  entry) -> reopen and add a guarded writer.
- If hook-trust review is removed and hooks run untrusted-by-default with no plugin-side opt-in -> reopen.
