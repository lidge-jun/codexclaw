## Goal (one slice)
Detect active goal mode by READING codex's native goal state: a read-only lookup
of `goals_1.sqlite` keyed by `thread_id` (= the hook payload `session_id`).

## Why now / dependencies
L8 defines that interview state must not mutate in goal mode. L10 defines the frozen spec consumed by goal-mode PABCD. Q-GM-1-f is RESOLVED: source = native goal DB read (not a self-owned marker).

## Scope (decision-complete)
- Files to add/edit: `plugins/codexclaw/components/pabcd-state/src/goal-active.ts` (NEW), `plugins/codexclaw/components/pabcd-state/src/hook.ts`, `plugins/codexclaw/components/pabcd-state/src/cli.ts`, `plugins/codexclaw/components/pabcd-state/test/goal-active.test.ts` (NEW), `plugins/codexclaw/components/pabcd-state/test/hook.test.ts`, `plugins/codexclaw/components/pabcd-state/test/cli.test.ts`.
- `goal-active.ts` resolves the codex goals DB path under `$CODEX_HOME` (`state/src/lib.rs:82` = `goals_1.sqlite`), opens it READ-ONLY, and looks up the row by `thread_id` (PK; `goals_migrations/0001_thread_goals.sql:2`).
- Returns one of: `active` (row exists with `status = Active`), `inactive` (no DB / no row / non-active status), `unreadable` (DB present but row unparsable).
- Active statuses considered "goal mode" = `Active` only; `Paused`/`Blocked`/`Complete`/budget/usage-limited are NOT goal-mode-active for interview-suppression purposes (a paused/complete goal should not block interview).
- `thread_id` comes from the hook payload `session_id` field (no goal field exists in payloads; `codex-rs/hooks/src/schema.rs:273/540/561`).
- The lookup is codexclaw-owned READ-ONLY; it never writes to the codex DB.
- Advisory skill text remains as the second defensive layer.
- Must-NOT-Have: no self-built goal store, no `.codexclaw/goal-active` marker, no WRITE to codex's DB, no prompt-only hard-deny claim, no silent ignore of an unreadable row (fail closed).

## IPABCD micro-cycle
- I: Not applicable in active goal mode; detected goal-active suppresses interview.
- P: Specify the read-only DB lookup helper + status mapping.
- A: Security reviewer checks DB-missing, row-missing, unparsable-row, paused/complete, and locked-DB cases.
- B: Implement `goal-active.ts` read-only lookup + CLI diagnostics.
- C: Run hook/CLI tests against a synthetic `goals_1.sqlite` fixture for each status.
- D: Done = the read-only DB lookup gives the PreToolUse hook a deterministic goal-active source.

## Acceptance (1-3 testable criteria)
1. A fixture `goals_1.sqlite` with an `Active` row for the session's thread_id makes goal mode active; a `Complete`/absent row makes it inactive.
2. A present-but-unparsable goal row yields `unreadable` and the caller fails closed (treats as active for suppression).
3. CLI diagnostics report the resolved DB path, the looked-up thread_id, parse status, and active/inactive interpretation.
