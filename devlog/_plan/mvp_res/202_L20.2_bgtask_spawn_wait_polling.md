# L20.2 (Decade 202) -- bgtask to spawn/wait Polling

Status: ANALYZED
Cluster: 3 · Phase: expansion · Shorthand: cxc
Source-of-record: 200_L20_clijaw_command_mapping.md; 260629_codexclaw_mvp/090_clijaw_command_mapping.md; 090.1 J-2

## Goal (one slice)
Map cli-jaw `bgtask` to Codex in-turn polling with `spawn_agent` and
`wait_agent`, with short-lived terminal process polling as a separate shell case.
Durable server-owned re-invocation is explicitly deferred to Phase 3 scheduler
work, not approximated in MVP.

## Why now / dependencies
- Upstream: L5 supplies inline subagent roles and Codex multi-agent tools.
- J-2 resolved that codexclaw has no server-owned durable bgtask equivalent.
- Downstream: L18 ultraresearch and L21 role ops can use subagent polling without
  promising a cli-jaw-style restart-durable background task.

## Scope (decision-complete)
Files to add/edit in implementation:
- `plugins/codexclaw/docs/command-mapping.md` or equivalent reference
- Role/subagent guidance that mentions long-running work
- Future L29 scheduler docs, when Phase 3 begins

Exact behavior:
- For agent work: spawn scoped subagents, then call `wait_agent` until completion
  before ending the turn.
- For terminal processes: use `exec_command` with PTY/session output, then poll
  with the matching stdin/poll mechanism until completion or user intervention.
- No work is considered safely in-flight after the main turn ends unless a later
  durable scheduler owns it.
- Phase 3 (L29/L30) owns true periodic or durable background jobs.
- Deep research that uses subagents is still in-turn orchestration; it is not a
  server-registered bgtask.
- User-facing docs should explain that `cxc` does not offer `bgtask add/watch` in
  MVP.

Must-NOT-Have:
- No `cxc bgtask` daemon or job store.
- No promise of completion callback after process exit.
- No hidden dependency on cli-jaw server, jaw bgtask, or web-AI sessions.
- No durable scheduling until L29 selects OS scheduler vs daemon.
