# L19.1 (Decade 191) -- Skill Exposure Mechanism

Status: BLOCKED(L12-L17)
Cluster: 2 (part B) · Phase: expansion · Shorthand: cxc
Source-of-record: 190_L19_skill_hub_rewrite.md; 260629_codexclaw_mvp/100_skill_hub.md; 090.1 J-6/J-11

## Goal (one slice)
Specify how codexclaw exposes skills without building a runtime hub engine:
only `dev` renders automatically, implicit-off skills remain hidden but
explicitly loadable, and `skill-hub` is the on-demand bootstrap catalog that
tells the agent what exists beyond `dev`.

## Why now / dependencies
- Upstream: L12-L18 must define final skill names and policies before the catalog
  can be accurate.
- J-11 is resolved through non-implicit bootstrap: `skill-hub` remains
  explicit-loadable and can also be reached by `dev` hub routing, without joining
  the default implicit set.
- Downstream: all on-demand skills depend on this mechanism to stay reachable
  while keeping the prompt small.

## Scope (decision-complete)
Files to add/edit after unblock:
- `plugins/codexclaw/skills/skill-hub/SKILL.md`
- `plugins/codexclaw/skills/skill-hub/agents/openai.yaml`
- `plugins/codexclaw/skills/skill-hub/references/catalog.md`
- `plugins/codexclaw/skills/*/agents/openai.yaml`

Exact behavior:
- `allow_implicit_invocation: true` means the skill is eligible for automatic
  rendering in `<skills_instructions>`.
- `allow_implicit_invocation: false` removes the skill from the automatic list
  but keeps explicit `$skill` or SKILL.md path mention working.
- `enabled=false` in config disables both implicit and explicit paths; on-demand
  skills must not use that route.
- `skill-hub` is on-demand and explains the hidden catalog: read
  `references/catalog.md`, find the needed skill, then explicitly load that skill.
- The hub is prose plus catalog. It does not inspect the filesystem at runtime,
  mutate Codex config, or dynamically inject skills.
- Each catalog row should include skill name, path, category, load_when trigger,
  implicit policy, and any codex-native gap.

Must-NOT-Have:
- No runtime skill loader or MCP service for skill discovery.
- No frontmatter `keywords`, `search_terms`, or `default_prompt` fields.
- No policy value in SKILL.md frontmatter; policy lives in `agents/openai.yaml`.
- No `disabled` on any on-demand skill that must remain explicitly loadable.

## IPABCD micro-cycle
- I: not interview-bearing.
- P: once L12-L18 are done, enumerate actual skill dirs and write the catalog from
  current files, not from stale plan names.
- A: reviewer checks the two-path model: implicit list is small, explicit mention
  still works for every hidden skill.
- B: author hub SKILL.md, add hub policy file, and update every skill policy to
  match catalog classification.
- C: `codex debug prompt-input` verifies visible set; explicit-mention smoke test
  verifies one hidden skill injects.
- D: done = hub points to a complete catalog and no hidden skill is disabled.

## Acceptance (1-3 testable criteria)
1. `skill-hub/agents/openai.yaml` has `allow_implicit_invocation: false` and remains explicit-loadable.
2. Each catalog entry has a matching skill directory and policy file.
3. A hidden on-demand skill is absent from implicit render but explicit-loadable.

## QA channel (node:test path / CLI stdout / tmux / data dump)
- node:test comparing catalog rows to actual skill directories and policy flags.
- `codex debug prompt-input` dump for rendered implicit list.
