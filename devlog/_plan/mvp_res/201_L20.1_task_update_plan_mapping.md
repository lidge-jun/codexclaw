## Goal (one slice)
Replace cli-jaw persistent `task` semantics with Codex-native `update_plan` only.
Codexclaw does not build a file-backed task database, owner assignment, ordering
DAG, or dashboard-visible task list in MVP.

## Why now / dependencies
- Upstream: no code dependency beyond Codex runtime's existing plan tool.
- J-1 resolved the contradiction: cli-jaw task features are intentionally dropped
  rather than recreated.
- Downstream: keeps L20 CLI surface small; no `cxc task` command is introduced.

## Scope (decision-complete)
Files to add/edit in implementation:
- `plugins/codexclaw/docs/command-mapping.md` or equivalent shipped reference
- Any `cxc doctor` mapping output that explains unsupported cli-jaw features

Exact behavior:
- cli-jaw `task add`, `task done`, `task start`, `task list`, `task assign`, and
  `--after` are mapped to no new codexclaw command.
- Agents should use Codex `update_plan({ explanation?, plan })` for per-turn work
  tracking when a task has multiple steps or benefits from visible sequencing.
- Plan statuses are `pending`, `in_progress`, and `completed` only.
- At most one plan item is `in_progress` at a time.
- The plan is operational guidance in the current session, not durable project
  state and not a cross-agent assignment system.
- Long-lived durable work belongs to goal files or future scheduler loops only if
  a later loop explicitly adds them.

Must-NOT-Have:
- No `cxc task` subcommand.
- No `.codexclaw/tasks.json` or dashboard integration.
- No owner assignment to roles or subagents via task state.
- No claim that `update_plan` persists across sessions.
