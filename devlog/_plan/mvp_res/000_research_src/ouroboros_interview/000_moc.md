# ouroboros / Q00 — Interview 하드닝 리서치 (MOC)

> 목적: Pass 8(Interview 하드닝) 설계의 ground truth 수집. ouroboros(Q00, "Agent OS")의
> Interview→Seed 루프와 "Nine Minds" 페르소나 모델, 그리고 그 **Codex 플러그인 / codex
> subagent fan-out** 구조를 실제 소스에서 확인한다. 결론은 MVP 폴더의 Pass 8 계획 문서로 환류한다.

상위 계획 문서(MVP): [080_pass8_interview_hardening_plan.md](../260629_codexclaw_mvp/080_pass8_interview_hardening_plan.md)

## 왜 이 리서치가 필요한가
- gjc(cli-jaw) = **서브에이전트가 질문을 만드는** 구조 (모순점 → 질문).
- jawcode = **메인이 직접 질문하는** 구조.
- ouroboros = 그 둘의 원류. 페르소나(Contrarian/Socratic/Ontologist)가 모순·가정·누락을
  캐고 그것을 질문으로 변환. codexclaw는 codex 네이티브 `spawn_agent`가 있으니 MCP 서버 없이
  각 Mind를 서브에이전트 role로 매핑 가능 → ouroboros의 host-driven fan-out이 공짜로 따라옴.

## 확인해야 할 것 (실 소스 검증 — 아래는 웹검색 기반 가설, untrusted)
1. 루프 단계: Interview → Seed → Execute → Evaluate → Evolve (정확한 단계명/전이 조건).
2. "Nine Minds" 페르소나 정의 형식 — 파일 스키마, on-demand 로딩 방식, Contrarian/Socratic/Ontologist의 실제 프롬프트.
3. Interview가 in-process인지, MCP 서버가 첫 authoring 질문을 소유하는지.
4. Codex 런타임 연동: `--runtime codex` 경로에서 인터뷰는 누가 소유하나(호스트 vs codex).
5. **codex subagent fan-out** 디스패치 코드 — 호스트 루프가 codex 서브에이전트를 병렬로 어떻게 돌리나.
6. ouroboros **codex 플러그인** 디렉토리 구조 (hooks/skills/plugin.json) — codexclaw가 차용할 패턴.
7. ⭐ **기억 vs 직접 수정**: ouroboros는 모순을 "기억"만 하는가? codexclaw는 서브에이전트가
   devlog/플랜을 **직접 편집**하며 진행하려 함 → ouroboros엔 없는 부분. 어떤 단계가 가장 가까운지 확인.
8. ⭐ **auto 모드**: ouroboros Evolve/자율 실행 트리거가 있나? 유저 개입 최소로 모순 도출→플랜 수정까지
   자율 진행하는 패턴. cli-jaw goal 모드 자율성과 비교 대상.
