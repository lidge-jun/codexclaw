# L29.1 (Decade 291) -- Scheduler Mechanism

Status: BLOCKED(Q-P3-1)
Cluster: 5 - Phase: 3 - Shorthand: cxc
Source-of-record: 290_L29_scheduler_mechanism_cli.md, 260629_codexclaw_mvp/040_phase3_overview.md

## Goal (one slice)
Select and implement the scheduler mechanism boundary: recurring work is driven
by `codex exec` launched from an OS scheduler, unless Q-P3-1 chooses a daemon.

## Why now / dependencies
- Upstream: depends on L6 install/activation so codexclaw is active when a new
  `codex exec` session starts.
- Downstream: gives L29.2 the writer target and gives L30 a predictable run
  invocation contract.

## Scope (decision-complete except Q-P3-1)
- Files to add/edit:
  - scheduler writer module under the existing `cxc` CLI surface
  - tests for launchd/cron rendering without installing system entries
  - docs or CLI help text for generated scheduler artifacts
- Exact behavior:
  - Generate a command that runs `codex exec "<prompt>"` from the target project.
  - Prefer launchd on macOS and cron or system timer only as explicitly scoped.
  - Keep the scheduler external to Codex hooks; hooks fire per session, not on a
    timer.
  - Provide dry-run output before any OS scheduler write.
- Must-NOT-Have:
  - No built-in cron loop in codexclaw.
  - No hidden always-on service in the default MVP.
  - No channel delivery or result forwarding in this pass.

## IPABCD micro-cycle
- I: not interview-bearing; Q-P3-1 is the scheduler fork.
- P: define writer adapter interface, generated `codex exec` command, dry-run
  output, and removal artifact shape.
- A: audit angle = "does scheduling stay outside the Codex runtime?" Reviewer
  checks no timers in hooks and no daemon process assumptions.
- B: implement writer rendering and dry-run mechanism for the chosen OS path.
- C: run node:test for rendered artifacts and inspect `cxc schedule add --dry-run`
  output.
- D: done = scheduler mechanism can install or preview recurring `codex exec`
  invocations once Q-P3-1 is resolved.

## Acceptance (1-3 testable criteria)
1. Dry-run output shows the exact `codex exec` command.
2. Writer tests render scheduler artifacts without touching the system.
3. No hook is used as a timer.

## QA channel (node:test path / CLI stdout / tmux / data dump)
- node:test for launchd/cron rendering.
- CLI stdout: scheduler dry-run artifact and generated command.

## Commit unit (one atomic conventional commit)
`feat(schedule): render os scheduler commands`

## Blocked-on (jun decision id, if any)
BLOCKED(Q-P3-1): OS scheduler vs always-on daemon.

Options:
- Option A - OS scheduler with `codex exec` (recommended). Impact: smallest MVP
  and matches 040 feasibility.
- Option B - always-on daemon using app-server/exec-server. Impact: more service
  lifecycle and experimental runtime surface.
- Option C - store-only planner. Impact: safest but not heartbeat-style UX.
