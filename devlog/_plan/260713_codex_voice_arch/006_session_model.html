Codex Session Model

FEM(음성 모델)은 전역이 아니라 per-thread로 활성화된다

🔑 핵심 답변

FEM(Frontend Model)은 전역 상주 모델이 아닙니다.
각 Thread가 thread/realtime/start를 호출할 때 해당 Thread에만 WebRTC 세션이 생성되고, 그 Thread 안에서만 FEM ↔ Orchestrator handoff가 작동합니다.
다른 Thread는 여전히 텍스트 전용으로 동작합니다.

Realtime Session Lifecycle — Thread A 기준

1. Thread 생성

thread/start — 일반 텍스트 Thread로 시작. realtime 없음.

2. Realtime 활성화 핵심

클라이언트가 thread/realtime/start 호출
→ 이 Thread에만 WebRTC AVAS 세션 생성 #27720
→ SDP 교환 → FEM 모델 연결
→ RealtimeConversationManager 인스턴스 생성 #12268

3. 이중 모델 동작

FEM: 음성 수신/발화 (이 Thread에만 바인딩)
Orchestrator: 코딩 작업 실행 (기존 codex-core Session)
둘 사이를 handoff protocol이 연결

4. 듀러블 재연결

네트워크 끊김 → Thread의 realtime 오케스트레이터가 새 Session으로 재연결 가능 #28826
UUIDv7 턴 ID로 중복 방지
appendText로 이전 세션 오버랩 재생 #28836

5. 세션 종료

WebRTC 연결 해제 → FEM 해제 → Thread는 텍스트 전용으로 복귀
다른 Thread에는 아무 영향 없음

비교: 기존 STT vs Realtime Voice 세션 모델

⌨️ 기존 STT 입력

  • 세션 모델: 없음 (stateless)
  • 클라이언트 측 변환 후 텍스트로 전송
  • 서버에 음성 관련 상태 없음
  • 모든 Thread에서 사용 가능
  • 별도 모델 인스턴스 없음

🎙️ Realtime Voice

  • 세션 모델: per-thread (stateful)
  • 서버에 WebRTC 세션 + FEM 바인딩
  • Thread별 독립 라이프사이클
  • 명시적 start/stop 필요
  • FEM + Orchestrator 이중 모델 구동
  • 듀러블 재연결 지원
