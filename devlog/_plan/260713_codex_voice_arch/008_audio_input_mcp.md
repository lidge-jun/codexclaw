# 008. MCP 라우팅 Audio Input (#22679)

## 연구 범위

1. 이 문서는 [#22679](https://github.com/openai/codex/pull/22679)의 오디오 입력 변경을 기록한다.
2. 핵심 질문은 MCP 도구 결과로 반환된 오디오 파일이 Codex의 텍스트 턴에 어떻게 들어가는가이다.
3. 이 변경은 실시간 음성 대화 기능을 추가한 것이 아니다.
4. 따라서 이 문서는 Realtime 세션, WebRTC, 마이크 캡처의 설계 문서가 아니다.
5. 분류상 이 기능은 텍스트 턴 경로 ①에 속한다.
6. 실시간 Voice Path ②와는 입력의 수명, 운반 방식, 모델 호출 경계가 다르다.

## 변경 요약

7. [#22679](https://github.com/openai/codex/pull/22679)은 `InputModality::Audio`를 추가한다.
8. 모델이 받을 수 있는 입력 modality 집합에 `Audio`를 표현할 수 있게 된다.
9. MCP 도구의 반환값 중 오디오 콘텐츠 블록을 Responses API 호환 구조로 변환한다.
10. 변환된 결과는 일반 문자열로 납작하게 합쳐지지 않는다.
11. 대신 구조화된 입력 콘텐츠로 보존되어 모델 호출에 전달된다.
12. 오디오 관련 function-call output item도 지원 대상에 포함된다.
13. 이 output item은 모델이 오디오 입력을 처리하는 호출 결과를 표현하는 경계다.
14. MCP 서버가 녹음 파일을 반환하는 경우가 대표적인 사용 사례다.
15. 단, 최종 모델이 오디오 입력을 지원해야만 해당 파일을 전달할 수 있다.
16. 지원하지 않는 모델에 오디오를 조용히 제거하거나 텍스트로 가장하지 않는다.
17. 대신 사용자가 원인을 알 수 있는 명확한 오류를 받도록 게이트한다.

## 처리 흐름

18. MCP 서버가 도구를 실행하고 녹음 파일을 결과 콘텐츠로 반환한다.
19. Codex의 MCP 결과 처리 계층이 콘텐츠 블록의 종류를 판별한다.
20. 오디오 블록은 Responses 입력에서 표현 가능한 오디오 구조로 매핑된다.
21. 이때 파일의 오디오 데이터와 관련 메타데이터가 구조화된 경계 안에 남는다.
22. 현재 모델의 input modalities를 확인한다.
23. 모델의 modality 목록에 `Audio`가 있으면 입력 변환을 계속한다.
24. `Audio`가 없으면 모델 호출 전에 명시적 오류를 반환한다.
25. 이 순서는 모델이 이해하지 못하는 콘텐츠를 전송하는 상황을 방지한다.
26. 오디오 입력이 허용된 뒤에만 Responses-compatible item으로 요청을 구성한다.
27. 구성된 요청은 일반 텍스트 메시지와 함께 하나의 텍스트 턴에 포함될 수 있다.
28. 따라서 사용자는 MCP 도구가 반환한 녹음 파일을 후속 질문의 근거로 사용할 수 있다.
29. 그러나 파일을 받는 순간 별도의 상시 음성 세션이 생성되는 것은 아니다.
30. 도구 호출이 끝나고 모델 요청이 끝나면 이 오디오 입력의 수명도 해당 턴에 종속된다.

## Code mode 계약

31. 코드 모드에는 엄격한 `audio(...)` 헬퍼가 추가된다.
32. 이 헬퍼는 오디오 입력을 코드 모드 문법 안에서 명시적으로 표현한다.
33. `audio(...)`는 임의의 문자열을 오디오로 추측하는 편의 문법이 아니다.
34. strict 계약을 사용하므로 허용된 인자와 콘텐츠 형태를 벗어난 입력은 실패한다.
35. 엄격한 헬퍼는 텍스트와 바이너리성 콘텐츠의 경계를 코드에서 분명하게 만든다.
36. 이는 MCP 결과가 후속 모델 입력으로 변환되는 시점을 추적하기 쉽게 한다.
37. 또한 지원하지 않는 모델에서 늦은 API 오류가 발생하는 것을 줄인다.
38. 코드 모드 사용자는 `audio(...)`를 실시간 마이크 입력 함수로 해석하면 안 된다.
39. 헬퍼가 나타내는 것은 이미 확보된 오디오 콘텐츠의 입력 표현이다.
40. 즉, 캡처 장치의 열기나 재생 장치의 제어는 이 PR의 계약에 포함되지 않는다.

## 스키마와 프로토콜 영향

41. 변경 과정에서 app-server 프로토콜 스키마가 재생성된다.
42. 재생성된 스키마는 오디오 관련 입력 및 output item의 타입을 반영한다.
43. 이 단계는 런타임 변환 코드만 고치는 것보다 중요하다.
44. app-server 클라이언트가 구조화된 오디오 콘텐츠를 이해하려면 공개 프로토콜도 함께 맞아야 한다.
45. 스키마 재생성은 수동으로 유사 타입을 추가하는 방식보다 drift를 줄인다.
46. 프로토콜 소비자는 새 타입을 알 수 없는 구버전과의 호환성을 별도로 고려해야 한다.
47. 다만 이 PR의 본질은 app-server에 realtime 음성 제어 명령을 추가하는 것이 아니다.
48. 새 타입은 MCP 결과에서 Responses 입력으로 이어지는 데이터 계약을 설명한다.

## 예시

49. 예를 들어 MCP 서버가 회의 녹음 파일을 반환한다고 가정한다.
50. 서버의 도구 결과에는 오디오 콘텐츠 블록이 포함된다.
51. 사용자가 “이 녹음의 핵심 결정을 요약해줘”라고 후속 요청을 보낸다.
52. 대상 모델이 `Audio` modality를 지원하면 파일은 구조화된 오디오 입력으로 전달된다.
53. 모델은 텍스트 요청과 녹음 입력을 같은 텍스트 턴에서 해석할 수 있다.
54. 대상 모델이 `Audio`를 지원하지 않으면 Codex는 지원 불가 오류를 반환한다.
55. 이 경우 파일이 무음으로 버려지거나 임의의 음성 인식 결과로 대체되지 않는다.
56. 사용자는 오디오 지원 모델로 전환하거나 MCP 결과를 다른 형태로 제공해야 한다.

## Realtime Voice와의 분리

57. 이 기능은 WebRTC 연결을 만들지 않는다.
58. 마이크 권한을 요청하지 않는다.
59. 오디오 장치를 선택하지 않는다.
60. 스피커 재생 스트림을 열지 않는다.
61. FEM, AVAS, handoff, durable realtime session을 활성화하지 않는다.
62. 그러므로 [#28856](https://github.com/openai/codex/pull/28856)에서 정리된 AVAS v1 방향과 직접 결합되는 기능이 아니다.
63. 실시간 경로의 세션 lifecycle은 [#27114](https://github.com/openai/codex/pull/27114) 및 관련 PR들의 관심사다.
64. MCP Audio Input은 그 세션 lifecycle 바깥에서 한 번의 모델 입력을 구성한다.
65. 두 경로를 “오디오를 다룬다”는 이유만으로 하나의 Voice 기능으로 묶으면 안 된다.

## 아키텍처 판정

66. [#22679](https://github.com/openai/codex/pull/22679)은 파일 기반 오디오 입력의 표현력을 높인다.
67. 사용자 음성 대화를 CLI나 데스크톱에 추가했다는 근거로 사용할 수 없다.
68. 현재 보이스 연구에서 이 PR을 인용할 때는 “MCP-routed audio file input”이라고 명시해야 한다.
69. 기능 경계는 MCP 도구 결과 → Responses-compatible structured input이다.
70. 기능 경계에 마이크 캡처 → WebRTC → voice model 응답은 포함되지 않는다.
71. 후속 조사에서는 모델 capability check가 호출 전 어느 계층에서 수행되는지 확인해야 한다.
72. 동시에 스키마 재생성 결과와 클라이언트 소비 여부를 별도로 점검해야 한다.
73. 이 문서의 결론은 오디오 입력 지원과 realtime 음성 지원을 분리해 기록하는 것이다.
74. 텍스트 턴 경로에 오디오 파일이 들어올 수 있다는 사실만으로 Voice Path가 개방된 것은 아니다.
