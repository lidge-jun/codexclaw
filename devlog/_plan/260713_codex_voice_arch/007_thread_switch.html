Thread Switch 시 Voice Session 동작

음성 세션 사용 중 다른 Thread로 전환하면 어떻게 되는가

🔑 핵심

UI는 한 번에 하나의 Thread만 표시합니다 (Active Thread Exclusivity, #14018).
마이크/스피커는 물리적으로 1개이므로, Thread를 전환하면 현재 음성 세션은 중단/일시정지됩니다.
하지만 백엔드의 듀러블 오케스트레이터는 살아 있어서, 돌아오면 재연결 + 오버랩 재생으로 이어갈 수 있습니다.

⚠️ 물리적 제약: 마이크는 1개

WebRTC 오디오 스트림은 마이크 입력과 스피커 출력을 점유합니다. 두 개의 Thread가 동시에 마이크를 사용할 수 없으므로, Thread 전환 = 현재 오디오 세션 중단은 필연적입니다.
서버 측 realtime orchestrator는 오디오 없이도 유지될 수 있지만, 실질적인 음성 대화는 불가능해집니다.

상황: Thread A에서 음성 대화 중

🎙️ Thread A Active + Voice ON

WebRTC Session ✅ AVAS v1 · 오디오 스트림 활성
마이크 캡처 중 · 스피커 출력 중
FEM (Frontend Model) ✅ 사용자 음성 수신 · 실시간 발화
Codex Orchestrator ✅ handoff 대기 중
UI Event Receiver ✅ 이 Thread의 이벤트를 UI에 렌더링 중
현재
상태

💬 Thread B Idle

codex-core Session 텍스트 전용 · 대기 중
WebRTC / FEM ❌ 없음
UI Event Receiver 이벤트 버퍼에 쌓이는 중 (렌더링 안 됨)
사용자가 Thread B를 클릭

1. UI Event Receiver 전환 핵심

Thread A의 mpsc::Receiver를 ThreadEventChannel.receiver에 반환.
Thread B의 Receiver를 select! 루프에 장착. #14018
→ 화면이 Thread B로 전환됨.

2. WebRTC 오디오 세션 중단

마이크 캡처 해제 · 스피커 스트림 정지.
클라이언트가 오디오 디바이스를 놓으면 FEM과의 실시간 대화 불가.
WebRTC 연결 자체는 transport 레이어에서 close 또는 idle 상태로 전환.

3. 서버 측 Orchestrator는 유지

Thread A의 codex-core Session과 듀러블 realtime orchestrator는 서버에 살아있음. #28826
진행 중인 턴이 있으면 완료까지 실행 (도구 호출, 코드 수정 등).
이벤트는 Thread A의 버퍼(ThreadEventStore)에 쌓임.

4. Thread B 활성화

Thread B의 이벤트 스트림이 UI에 렌더링 시작.
텍스트 입력 가능. 필요 시 이 Thread에서도 realtime/start 가능
(새 WebRTC 세션이 Thread B에 바인딩됨).

전환 직후 상태

🎙️ Thread A Voice Suspended

WebRTC Session ⏸️ 연결 close/idle · 오디오 없음
FEM ⏸️ 음성 입출력 불가 · 대화 중단
Codex Orchestrator ✅ (서버) 듀러블 · 진행 중 턴 계속 실행
결과는 버퍼에 저장
UI 이벤트 버퍼링 (화면에 안 보임)
전환

💬 Thread B Active (Text)

codex-core Session ✅ 텍스트 입출력 활성
WebRTC / FEM ❌ 없음 (시작 안 했으므로)
UI Event Receiver ✅ 이 Thread의 이벤트 렌더링 중
Thread A로 다시 돌아올 때

1. UI Receiver 복원

Thread A의 Receiver를 다시 select! 루프에 장착.
버퍼에 쌓였던 이벤트들이 UI에 리플레이됨.

2. Realtime Session 재연결 #28826

듀러블 orchestrator가 새 Session 인스턴스로 재연결.
UUIDv7 턴 ID로 이전 세션과 중복 방지.
WebRTC SDP 재교환 → 마이크/스피커 재점유.

3. 세션 오버랩 재생 #28836

appendText (assistant role)로 이전 세션의 마지막 대화를
conversation item으로 재생 → FEM이 맥락을 이어받음.
사용자에게는 대화가 끊기지 않은 것처럼 보임.

4. 음성 대화 재개

FEM이 다시 마이크 입력 수신 시작.
→ 사용자 관점에서는 "잠깐 나갔다 돌아온" 느낌.

🤔 만약 Thread B에서도 음성을 켜면?

가능합니다. Thread B에서 thread/realtime/start를 호출하면 새로운 WebRTC 세션이 Thread B에 바인딩됩니다.
이 경우 Thread A의 세션은 완전히 해제되고, 마이크/스피커가 Thread B로 이동합니다.
다시 Thread A로 돌아가면 듀러블 재연결이 Thread A의 orchestrator와 새 WebRTC 세션을 만듭니다.
