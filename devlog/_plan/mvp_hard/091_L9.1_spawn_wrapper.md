# L9.1 / 091 — Production Spawn Wrapper (runtime impl)

Status: P (plan) · 2026-06-30 · mvp_hard loop L9.1 · class C3 (subagent/model runtime)

> L9 (090) closed the parity PLAN. This sub-loop ships the runtime piece it deferred:
> a production wrapper that turns `resolveSpawnConfig()` + a role TOML into a concrete
> Codex `spawn_agent` payload. No new first-class agent roles, no global config mutation.

## Why

`090_L9` documents the gap precisely: the `.codexclaw/subagents.json` store, MCP/GUI
roundtrip, and `resolveSpawnConfig(cwd, role)` resolver all ship and are tested, but
**no production code consumes the resolver when calling Codex `spawn_agent`**. So a
configured per-role model/prompt override never reaches a live subagent launch. This
sub-loop closes exactly that, and only that.

## Ground Truth (read before edit)

- Roles + agent_type mapping: `plugins/codexclaw/agents/README.md` — explorer->`explorer`,
  reviewer->`explorer`, executor->`worker`. Built-in types only (no plugin role registration).
- Role prompt source: `plugins/codexclaw/agents/{explorer,reviewer,executor}.toml`,
  `developer_instructions = """..."""` triple-quoted block + `model = "default"`.
- Resolver: `plugins/codexclaw/components/subagent-config/src/store.ts:resolveSpawnConfig`
  returns `{ role, model|null, usesMainModel, promptOverride|null }`.
- Inline-injection pattern (omo parity): `spawn_agent({ agent_type, message: "TASK: <role
  prompt + task>" })` — the README already documents this as the B-opt2 contract.

## Design (diff-level)

New module `plugins/codexclaw/components/subagent-config/src/spawn-wrapper.ts`:

1. `ROLE_AGENT_TYPE: Record<RoleName, "explorer" | "worker">` — the canonical mapping
   (`explorer`/`reviewer` -> `explorer`, `executor` -> `worker`).
2. `parseRoleToml(text)` — a NARROW field reader (not a general TOML parser): extracts
   `model` (simple `key = "value"`) and `developer_instructions` (triple-quoted block).
   Zero third-party deps (build.mjs requires node:* only). Returns
   `{ model: string|null, developerInstructions: string }`; missing/malformed -> safe
   defaults (empty instructions, model null).
3. `readRoleToml(agentsDir, role)` — read `<agentsDir>/<role>.toml`; missing file ->
   `{ model:null, developerInstructions:"" }` (never throws).
4. `buildSpawnPayload({ role, task, resolution, developerInstructions })` — PURE builder:
   - `agent_type` = `ROLE_AGENT_TYPE[role]`,
   - `message` = `TASK:`-prefixed composition of the effective role prompt + the task,
   - effective role prompt = `promptOverride ?? developerInstructions` (override replaces;
     never fabricates when both empty),
   - include `model` ONLY when `resolution.usesMainModel === false` and `model` is a
     non-empty string; default mode omits `model` so the subagent inherits the main model.
5. `resolveSpawnPayload(cwd, role, task, agentsDir)` — the production entry point that
   composes `resolveSpawnConfig` + `readRoleToml` + `buildSpawnPayload`.

### Invariants

- Default mode (`model:null`) => payload has NO `model` key (inherit main model).
- `promptOverride` (when set) REPLACES the TOML developer_instructions in the message.
- `agent_type` is always a built-in codex type; the wrapper never invents a role name.
- Pure builder: no IO in `buildSpawnPayload`; only `readRoleToml`/`resolveSpawnPayload`
  touch the filesystem, and both are total (never throw).

## Acceptance

| Check | Evidence |
|-------|----------|
| Model override applied | configured `model` role => payload.model === that id |
| Default inherits | default role => payload has no `model` key |
| Prompt override replaces | promptOverride set => message contains override, not TOML body |
| agent_type mapping | executor->worker, explorer/reviewer->explorer |
| TOML parse | reads `developer_instructions` triple-quoted body from a real role file |
| Total/fail-safe | missing TOML / malformed input => safe defaults, never throws |
