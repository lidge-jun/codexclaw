# L3.1 / 031 — Hook → `transition()` Wire (pass L3b, the missing wire)

Status: P (plan) · 2026-06-30 · mvp_hard loop L3 pass B · class C3 (persistence: writes phase + ledger)

> This is THE parity gap from the L1 audit: codexclaw has every FSM primitive but no
> code path where a user command actually moves `state.phase`. L3a shipped the parser;
> L3b connects it: a chat-submitted `orchestrate <verb>` now calls
> `readState → transition() → writeState → appendLedger` and injects the resulting
> phase directive. Chat submission = HUMAN source = free-pass (advisory; forward
> transitions are NOT attest-blocked here — that gate is the agent/CLI path in L4).

## Goal (L3b)

Wire `parseOrchestrateCommand` into `handleUserPromptSubmit` as the AUTHORITATIVE,
parser-first path. When a chat prompt carries an explicit orchestrate command:
1. parse it (L3a);
2. apply the transition to file state with a HUMAN free-pass (no attest requirement
   on forward edges; illegal *adjacency* is still refused — L2 invariant holds);
3. persist via `writeState` + append a `ledger.jsonl` entry;
4. inject the new phase's directive (or a status/reset response) as additionalContext.

The loose `detectTrigger` heuristic stays as the fallback ONLY when the strict parser
returns null.

## Human free-pass design (invocation-source split)

cli-jaw splits human vs agent by boss-token; codexclaw has none, so the split is the
**invocation source**, which the hook event already encodes:
- A **chat-submitted** command arrives via `UserPromptSubmit` → HUMAN → free-pass:
  forward transitions advance WITHOUT requiring `--attest`. If `--attest` is supplied
  it is still recorded in the ledger (and flips the gate flag honestly), but its
  absence does NOT block a human-driven forward move.
- An **agent/tool-invoked** command will arrive via the `cxc orchestrate` CLI (L4) →
  AGENT → gated: forward edges require `--attest` (reusing `transition()`'s built-in
  `validateAttest`). L3b does NOT implement the CLI; it only implements the human path.

Implementation: add a `human` free-pass mode to the transition application so the
forward-edge attest requirement can be bypassed for chat, WITHOUT weakening the L2
adjacency table. Because `transition()` itself hard-couples attest to forward edges,
L3b adds a thin `applyHumanTransition(state, verb)` helper in a new
`orchestrate-apply.ts` that: for forward edges, flips the unlocked gate flag directly
(the human asserts the phase is done) then runs `canEnter` adjacency; for `status`/
`reset` it returns the control result. This keeps `transition()` (agent-gated)
untouched and adds the human variant alongside it.

## Accept criteria (testable)

- After `handleUserPromptSubmit({prompt:"orchestrate p", ...})` the persisted state has
  `phase:"P"` and `ledger.jsonl` gained one entry with `to:"P"`.
- Human free-pass: `orchestrate b` from phase A advances to B WITHOUT `--attest`
  (auditPassed pre-flipped), while an illegal `orchestrate c` from phase P is refused
  (state unchanged, no ledger entry).
- `orchestrate reset` returns phase to IDLE and clears auditPassed/checkPassed.
- A prompt with no orchestrate command still triggers the loose `detectTrigger` path
  (no regression in existing hook tests).
- `npm test` green at 244+; `npm run build` idempotent (from repo root).

## Risk / rollback

- Risk: double-handling if both the parser and `detectTrigger` fire. Mitigation:
  parser-first EARLY RETURN; `detectTrigger` only on parser-null.
- Risk: human free-pass flipping gate flags could let a later agent path skip real
  evidence. Mitigation: the flag flips are scoped to the human chat transition and the
  ledger records `reason:"chat"`; the agent CLI path (L4) uses the un-weakened
  `transition()` with `validateAttest`. Document this boundary in the apply helper.
- Rollback: revert hook.ts wire + delete orchestrate-apply.ts; parser (L3a) stays.

## Audit verdict (A gate — independent reviewer, 2026-06-30)

Verdict: **PLAN OK with fixes**. Confirmed: pre-flip list is minimal (only `A->B`
needs auditPassed, `C->D` needs checkPassed; `P->A`/`B->C` need none) and
`transition()` must stay untouched (no actor param) so L4's agent-gating survives.
Folded into the build scope:

1. **HIGH — preserve goal-mode I suppression on the parser path**: the parser block
   must sit AFTER `if (turn && state.injectedTurns.includes(turn)) return "";`
   (hook.ts:179) and, before any state/ledger write, apply the SAME guard
   `if (cmd.verb === "I" && suppressesInterview(getGoalActiveStatus(session_id))) return "";`.
   Otherwise a strict `orchestrate i` re-opens Interview in goal mode (canEnter("I")
   is always allowed).
2. **MEDIUM — D->I flag clearing**: `D->IDLE` clears gate flags, but legal `D->I`
   does not. The human apply path must clear `auditPassed/checkPassed` when entering
   `I` (fresh cycle), and a test must assert flags are cleared after `D->I`. (Keeps a
   stale human-flipped checkPassed from leaking into the next cycle.)
3. **MEDIUM — ledger/reset idempotency**: record the turn in `injectedTurns` BEFORE
   returning so a same-turn re-fire does not double-append (appendLedger is a blind
   append). Make `reset` from IDLE a no-op (no ledger entry). Add tests: duplicate
   same-turn command appends exactly one ledger entry; reset-from-IDLE is a no-op.
4. **LOW — reset bypasses adjacency explicitly**: `reset` must NOT route through
   `canEnter` (L2 rejects mid-cycle `P/A/B/C->IDLE`). Instead it directly writes the
   `D->IDLE`-style cleared state (phase IDLE, flags reset, orchestrationActive false)
   and is documented as an explicit control override, not an adjacency edge.
