## Gap

- `handleStop` only blocks under an ACTIVE goal mid-cycle (hook.ts guard 2b). In an
  interactive Interview there is no active goal, so a pending interview question or an open
  high-severity contradiction does NOT keep the turn alive — the session can end with the
  interview unfinished.
- `minds.ts` (selectMinds) and `triage.ts` (triageContradiction) are pure helpers with NO
  runtime caller wiring the dispatch→triage→ask→record→rescan loop together.

## Design (diff-level)

### 1. DROPPED — Narrow I-phase Stop guard

Not implemented. The I-phase Stop guard had no valid domain (A-gate B2): under an active goal
the interview is suppressed (no `question_asked` ever recorded), and with no active goal a
Stop block would violate the L6 "interactive pauses for the human" invariant. `handleStop`
keeps its existing behavior unchanged — it never drives the Interview.

### 2. rescan-coordinator.ts (pure helper) — the only code this loop ships

- `hasPendingInterviewWork(cwd, sessionId, tracker)` → `{ pendingQuestionIds: string[];
  highContradictionCount: number; pending: boolean }`. Pending question = a `question_asked`
  event whose `(turnId, questionId)` pair has no matching `answer_recorded` in the same
  ledger (match on the PAIR, not `questionId` alone — A-gate B1: dimension-based ids are
  reused across rounds, so a `questionId`-only match would treat a new question as answered).
- `computeNextScanRound(tracker)` → next scan `roundId` from `tracker.scanRounds`/`roundId`.
  MONOTONIC RULE (A-gate B1): the next scan round is derived from `scanRounds` only —
  `scanRounds + 1` — NOT from `roundId` or `lastScanRoundId` (those are the interview turn
  horizon, a different counter). Conflating them is non-monotonic. Total: a malformed/missing
  `scanRounds` reads as 0 → next = 1.
- Documents the main-session sequence: select Minds (adaptive, cap 3) → dispatch read-only
  contradiction lenses → triage severity (high → ask_user; low/med → recorded assumption) →
  record answer (PostToolUse ledger) → rescan. The helper computes signals only; it never
  asks, writes state, or dispatches — the main session owns that (T4/T7).
- It is for the INTERACTIVE interview only (no active goal). It is never wired into
  `handleStop`; the main session calls it to decide whether to ask another question.
- CONTRADICTION SOURCE (A-gate B3): the high-severity contradiction signal is read ONLY from
  the in-memory `tracker.contradictions`. The helper NEVER reads the scan-evidence ledger
  (`readInterviewEvents`) — that reader is unfiltered and would absorb QA lines from the
  shared `.codexclaw/interviews/<id>.jsonl` file. Question pending is read via `readQaEvents`
  (which filters to question_asked/answer_recorded); contradictions come from the tracker.
- SESSION SCOPING (A-gate): the caller MUST pass the SAME `sessionId` used for the tracker
  and the ledger; a mismatched id silently yields "no pending". The helper documents this and
  the main session always reads both from its own session state.
- GOAL-ACTIVE GUARD (A-gate B3): `hasPendingInterviewWork` returns an EMPTY result
  (`pending:false`) when a goal is active, so even a future mis-wiring can never use it to
  resume an interview under a goal. It takes a `GoalActiveDeps`-style injectable status reader
  (default: real goals_1.sqlite read) for testability. This makes the "no interview during a
  goal" constraint hold at the helper boundary too, not just at the hook.

### 3. "No interview during a goal" boundary (explicit)

Make the existing invariant explicit and regression-guarded: when a goal is active,
`suppressesInterview` (active|unreadable) already blocks I-entry (UserPromptSubmit) and
hard-denies `request_user_input` (PreToolUse). Add a test asserting goal-active suppresses
interview, and a one-line note in `loop`/`pabcd` skills that goal mode is PABCD-only,
interview is HITL-only (no active goal). No new runtime branch — this pins behavior already
present so a future change can't silently arm an interview under a goal.

REGRESSION GUARD MUST BE REAL (A-gate B2): the guard test must inject an `openDb` stub that
returns a `status='active'` row and assert the interview path is actually suppressed — not
rely on a missing DB (which suppresses for the wrong reason and yields a false green). The
cleanest seam is `handlePreToolUseFailClosed(raw, deps)`, which already accepts
`GoalActiveDeps` (`deps.openDb`): assert that an active-goal stub denies `request_user_input`,
and that an inactive stub allows it. This proves the deny on the real status path.

## A-gate findings (adversarial review — must resolve before B)

B1 (hard blocker) — pending-question match key. The ledger event identity is
`(turnId, questionId, kind)`. If a later round reuses the same `questionId` (model picks
dimension-based ids like `goal`/`scope`), an old `answer_recorded` makes a NEW
`question_asked` look answered → silent under-block. Fix direction: match pending on
`(turnId, questionId)`, not `questionId` alone.

B2 (hard blocker) — removing the active-goal gate for `phase==="I"` breaks the L6 invariant
"a plain interactive session pauses for the human" (guard 2b, hook.ts:400; loop/SKILL.md:33).
In the canonical interactive interview (no goal), the model asks via `request_user_input` and
ends its turn; the I-branch would return `decision:"block"` and re-prompt the model instead
of yielding to the human. The branch arms continuation in exactly the no-goal case that must
NOT continue.

Deeper contradiction (the reason to stop and ask): the design's motivation — "drive the
interview to completion autonomously" — only applies under an ACTIVE goal. But when a goal is
active, `request_user_input` is hard-denied upstream (suppressesInterview = active|unreadable,
goal-active.ts:88), so no `question_asked` is ever recorded and the pending-question trigger
can never fire on the goal path. So:
- goal active  → interview suppressed → no pending question → guard has nothing to act on;
- goal inactive → interactive → guard MUST pause for the human (L6), not block.
Both domains reject the pending-question guard. The I-phase Stop guard as specified has no
valid domain. This is the contradiction surfaced back to the user.

Secondary: unknown contradiction severity reconstructs to `high` (interview.ts:109), so a
malformed tracker array would also trip the high-contradiction trigger; if the guard survives
redesign it must route through the single MAX_STOP_BLOCKS cap (no parallel return) and treat
`unreadable` as release (fail-safe).
