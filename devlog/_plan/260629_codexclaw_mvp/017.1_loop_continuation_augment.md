## 4. 017 PABCD loop 보강 제안

### 4.1 "Pass"를 작은 FSM item으로 승격

017의 pass map은 dependency order와 done criteria를 테이블로만 둔다. omo ulw-loop를 흡수하면 각 pass를 다음 필드가 있는 durable item으로 볼 수 있다:

```ts
type PabcdLoopItem = {
  id: string;
  phase: "I" | "P" | "A" | "B" | "C" | "D";
  status: "pending" | "active" | "blocked" | "failed" | "complete" | "needs_user_decision" | "superseded";
  dependsOn: string[];
  successCriteria: { id: string; scenario: string; expectedEvidence: string; status: "pending" | "pass" | "fail" }[];
  evidenceRefs: string[];
  blockerSignature?: string;
  supersededBy?: string[];
};
```

Rationale:
- `ulw-loop`의 `isCompletionBlocking()` / superseded replacement rule이 pass 재계획과 later accepted/appended story에 유용하다.
- 017의 "one atomic implementation unit = full P→A→B→C→D" 원칙은 유지하되, pass별 evidence/criteria를 ledger-first로 다룬다.

### 4.2 Stop continuation은 "phase auto-advance"가 아니라 "active item continuation"으로 제한

018.3은 Stop auto-advance를 위험하다고 봤고, 현재 `handleStop()`은 passive다. 따라서 보강 방향은:

- Pass 2/018.3의 UserPromptSubmit directive injection은 그대로 유지.
- 별도 후속 pass에서 `pabcd-continuation` Stop hook을 만들 경우, Stop에서 phase를 advance하지 않는다.
- Stop hook은 `.codexclaw/sessions/<session>.json` + `.codexclaw/ledger.jsonl` + active loop item을 읽어 "아직 active item의 required evidence/checklist가 남았는가?"만 판단한다.
- 남았으면 omo처럼 `decision: "block"`과 continuation directive를 출력한다.
- `stop_hook_active`, context-pressure transcript, missing state, no active orchestration, all criteria pass이면 silent.

이러면 018.3의 "Stop은 direct additionalContext가 불가능" 제약과 "Stop ledger spam" 회피를 동시에 만족한다.

### 4.3 checklist count 규칙을 codexclaw plan에도 명시

start-work-continuation의 실측 규칙을 017 보강 규칙으로 차용:

- Count 대상: 각 pass 문서의 `## TODOs`와 `## Final Verification Wave`의 column-0 checkbox만.
- Acceptance Criteria / Evidence / Definition of Done nested checkbox는 progress counter에서 제외.
- Stop continuation directive는 다음 미완료 top-level checkbox만 수행하게 한다.

효과:
- 자동 재개가 "문서 속 nested checklist"에 걸려 무한 loop 되는 것을 방지한다.
- `017`의 pass map을 실제 runnable checklist로 쪼갤 때 continuation semantics가 안정적이다.

### 4.4 Evidence ledger schema를 최소 확장

현재 codexclaw `LedgerEntry`는 phase transition 중심이다. PABCD loop continuation을 흡수하려면 transition 외 event를 기록할 수 있어야 한다.

제안:

```ts
type PabcdLedgerKind =
  | "phase_transition"
  | "item_started"
  | "criterion_passed"
  | "criterion_failed"
  | "item_blocked"
  | "item_completed"
  | "continuation_injected"
  | "gate_review";
```

필수 evidence fields:
- `itemId`, `criterionId?`, `phase`, `sessionId`, `commands`, `artifact`, `adversarialClasses`, `cleanup`, `sourceRefs`.

주의:
- 기존 `appendLedger()` export는 삭제하지 말고 overload/helper 추가.
- Unknown-key passthrough 금지 discipline은 유지한다.

### 4.5 PABCD의 B/C/D에 ultrawork completion contract 삽입

017의 Done 기준을 "tests green"에서 한 단계 보강:

- B: PIN/RED/GREEN discipline. Existing behavior 변경이면 characterization first.
- C: real-surface proof 필수. CLI/data-shaped면 auxiliary surface도 first-class.
- D: cleanup receipt + independent review/AdversarialVerify + ledger append 뒤에만 close.

LIGHT/HEAVY tier도 채택:
- LIGHT: narrow existing layer change, 1-2 criteria, one real-surface proof, self-review.
- HEAVY: new abstraction/security/external/schema/concurrency/cross-domain/care signal, 3+ criteria, independent reviewer loop.

## 5. 080 Interview 하드닝 보강 제안

080은 이미 ouroboros 기반으로 interview tracker, 5-Mind, severity-based auto mode, Interview→Goal freeze를 정의한다. ulw-plan을 흡수하면 다음을 추가할 수 있다.

### 5.1 CLEAR/UNCLEAR routing을 080의 interview entry에 명시

080의 `D5`는 "인터뷰는 goal 밖에서 끝나고 frozen spec을 남긴다"가 핵심이다. 여기에 ulw-plan routing을 붙인다:

- CLEAR: 사용자가 outcome을 알고 있다. 메인은 repo/docs explore 후 surviving owner-decision만 묻는다.
- UNCLEAR: outcome 자체가 fuzzy. 메인은 질문으로 책임을 넘기지 않고 best-practice default를 채택·공지한다.
- ON THE FENCE: CLEAR로 취급해 exactly one question.
- Explicit "interview me / ask me"는 CLEAR override.

080의 existing auto-mode와 충돌 없음:
- 080의 low/medium contradiction auto-default, high contradiction user escalation은 UNCLEAR의 "default except irreversible/destructive/safety-critical"와 동일 계열이다.
- Dialectic Rhythm Guard는 CLEAR path의 질문 남발 방지에도 쓸 수 있다.

### 5.2 Topology lock을 5-Mind loop 앞에 둔다

ulw-plan CLEAR path의 topology lock을 080에 흡수:

- 먼저 1-6 top-level components를 식별한다.
- 각 component는 독립적으로 success/fail 가능한 outcome 단위여야 한다.
- 5-Mind subagent는 "전체를 추상적으로 비판"하지 말고 component별 contradiction/gap만 반환한다.

효과:
- 080 D2의 "서브에이전트는 모순점만 도출" 원칙과 잘 맞는다.
- 질문 생성은 메인만 하므로 request_user_input 제약도 유지된다.

### 5.3 Approval gate를 Interview freeze 직전의 hard gate로 추가

080의 frozen spec 산출 직전에 ulw-plan approval gate를 추가:

- Draft/interview ledger에 `status: awaiting-approval`, pending action `freeze interview-spec`, approach/defaults/open owner-decisions를 기록.
- 사용자에게 brief를 한 번 제시한다.
- Approval은 **frozen spec 작성/확정만 승인**한다. Implementation/goal execution 승인이 아니다.
- Approval 없이 goal/PABCD로 넘어가지 않는다.
- `$start-work` bootstrap 같은 예외가 codexclaw에 생기더라도, 예외를 문서화한 단일 path로만 허용한다.

### 5.4 Frozen spec에 "Decisions I made for you" 블록 추가

UNCLEAR/default-heavy interview에서는 frozen artifact에 다음 블록을 둔다:

- Routing call: CLEAR/UNCLEAR/override 이유.
- Adopted defaults: default, rationale, reversibility, sourceRefs.
- Owner decisions asked: question, selected/default answer, evidence.
- Assumptions remaining: severity, closure mode (`safe_default` / `ledger_only` / `genuine-deadlock`).

효과:
- 080의 `assumptions` / contradictions tracker가 user-visible artifact로 이어진다.
- 잘못 UNCLEAR로 routing된 경우 사용자가 gate에서 한 줄로 veto 가능하다.

### 5.5 Interview readiness에 "zero executor judgment" 조건 추가

080의 `isInterviewReady()`는 다음을 만족해야 P로 갈 수 있게 보강:

- objective defined.
- scope IN/OUT explicit.
- approach decided.
- success criteria include exact scenario/tool/invocation/evidence path.
- no high-severity unresolved contradiction.
- all owner-decisions either answered or explicitly defaulted with reversibility.
- frozen spec has sourceRefs for repo/system facts.

이 조건은 ulw-plan의 "decision-complete; executor has zero interview context" 원칙을 codexclaw I→P gate에 옮기는 것이다.

## 6. 후속 구현 슬라이스 제안

이 문서는 직접 017/080을 수정하지 않는다. 반영 시 충돌을 줄이려면 아래 순서로 별도 pass/PR을 권장한다.

1. `pabcd-state`에 loop item/evidence types 추가: 기존 `State` strict reconstruct discipline 유지.
2. `pabcd-state` ledger event helper 추가: 기존 `LedgerEntry` transition API 보존.
3. `pabcd-continuation` Stop hook 신설: phase advance 금지, active item continuation만 block.
4. `pabcd` directive text 보강: plan+ledger first, first unchecked top-level checkbox, evidence+cleanup before done.
5. 080 implementation 시 interview routing/freeze approval gate 추가: CLEAR/UNCLEAR/topology lock/defaults block.
6. Tests: Stop anti-loop, context-pressure silence, no-active-state silence, active criteria continuation, all-done silence, I→P readiness with frozen spec.

## 7. 결론

- omo의 `start-work-continuation`은 codexclaw 018.3을 대체하는 것이 아니라, **Stop block 기반 active-work continuation** 패턴으로 별도 흡수해야 한다.
- omo의 `ulw-loop`은 codexclaw 017의 "many small PABCD loops"에 **durable goal/item FSM + evidence ledger + reconciliation discipline**을 더하는 참고 모델이다.
- omo의 `ultrawork`는 codexclaw B/C/D의 execution quality gate로, `ulw-plan`은 codexclaw I/P 및 080 interview hardening의 **explore-first + approval-before-freeze** 모델로 흡수하는 것이 가장 충돌이 적다.
