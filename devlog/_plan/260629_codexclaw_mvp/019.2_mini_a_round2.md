## Finding B — role pickup path (source-traced)
Verified in `core/src/config/agent_roles.rs`:
- User-defined roles load from TWO places per config layer:
  1. `[agents.<name>]` tables in config.toml (with optional `config_file` pointing to a .toml).
  2. `discover_agent_roles_in_dir(<layer.config_folder()>/agents/)` — scans an `agents/` dir for
     role `.toml` files.
- Built-ins (`explorer`, `awaiter`) are `include_str!`-compiled (role.rs:374-378).
- So role discovery is CONFIG-LAYER based, anchored at a layer's `config_folder()` + `/agents`.

### Open sub-question Q-PORT-1b (NOT fully resolved — honest gap)
- Whether an INSTALLED PLUGIN contributes a config layer whose `config_folder()/agents/` is scanned
  is NOT proven. Evidence is mixed:
  - core-plugins loader DOES interact with `ConfigLayerStack` (loader.rs).
  - BUT plugin.json's validated schema has NO `agents` field (only skills/hooks/mcpServers/interface)
    — so plugin-provided agent roles may NOT be auto-discovered the way skills/hooks are.
- Implication for T-025: do NOT assume `plugins/codexclaw/agents/*.toml` is auto-picked up.
  Two robust options:
  - B-opt1: codexclaw `enable` writes role entries into the user config layer
    (`~/.codex` `agents/` dir or `[agents.*]` in config.toml) — same controlled-write discipline as
    feature flags (027). Roles then load via the verified config-layer path.
  - B-opt2: spawn subagents WITHOUT a registered role, passing full instructions inline in
    `spawn_agent({message:"TASK:..."})` (omo's actual pattern — omo puts role/specialty INSIDE the
    message, see ulw-loop SKILL.md). This needs NO role-file pickup at all.
- RECOMMENDATION: Phase 1 uses B-opt2 (inline instructions, omo-proven) → unblocks T-025 with zero
  dependency on plugin role discovery. Keep agents/*.toml as the SOURCE for those inline prompts and
  as future role files (B-opt1) once pickup is confirmed.
