## Scope boundary

  - `src/config-features.ts` — pure functions: parse the `[features]` table from a config.toml string,
    compute the minimal edit to set the 4 declared flags true (only those not already true), apply the
    edit by LINE-level insertion/update (never reserialize the whole file → preserves all other lines
    byte-identical), and compute the inverse revert plan (restore prior value / remove keys codexclaw added).
  - `src/enable.ts` — orchestration: read config.toml, write timestamped `.bak`, apply the feature edit,

  - `package.json` mirroring pabcd-state.
- OUT: actually running enable against the real `~/.codex/config.toml` (tests use fixtures/temp dirs — do

## Key invariant (the "config-untouched guard")
Applying enable then uninstall to any config.toml yields a file byte-identical to the original. Applying
enable changes ONLY the declared `[features]` keys (and only those not already true); every other byte
unchanged. This is enforced by line-level editing (not TOML reserialization) + tested by string diff.

## Edge cases to handle (from a quick adversarial pass — confirm in A)
- No `[features]` table at all → create one (appended) without disturbing existing tables.
- Flag present but `false` → flip to true, record prior `false` for revert.
- Flag already `true` → no-op, NOT recorded as codexclaw-added (so uninstall won't remove a user's pre-existing true).
- Flag absent → add `key = true`, record as codexclaw-added (uninstall removes the whole line).
- Comments / blank lines / inline values elsewhere → must survive byte-identical.
- CRLF vs LF, trailing newline presence → preserve.
- Re-run enable twice → idempotent, manifest not double-counting.
