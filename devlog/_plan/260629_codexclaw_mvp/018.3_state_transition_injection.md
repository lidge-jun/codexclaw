# 018.3 — State-Transition-Time Prompt Injection Hardening (diff-level design)

Status: P (PLANNING)  ·  Phase 1 · builds on Pass 2 (018.2)  ·  research+design doc only (no src edits)
Scope: design how codexclaw auto-carries the new IPABCD phase directive on the NEXT turn after a phase
advance (e.g. P→A) WITHOUT the user re-typing a trigger. Grounded in jawcode 99.03 M2 + codex-rs hooks.

---

## 1. jawcode mechanism summary (ground truth, with citations)

jawcode (jwc, a Codex fork) solves "the next turn carries the new phase directive" with the
**99.03 M2 `pabcd-stage-context`** rail — a per-turn, compaction-immune stage header regenerated from
the state file every turn. This is the jwc port of cli-jaw's `getPrefix()`.

### 1.1 The per-turn re-read + re-inject (M2, the authoritative mechanism)

Every `prompt()` turn, `AgentSession` builds a hidden custom message and prepends it to the user
message. The header text is rebuilt from the state file each turn, so it survives compaction by
construction (it is never carried only in conversation history):

`packages/coding-agent/src/session/agent-session.ts:4787-4797`
```ts
async #buildPabcdStageMessage(): Promise<CustomMessage | null> {
    const cwd = this.sessionManager.getCwd();
    const sessionId = this.sessionManager.getSessionId();
    const result = await readPabcdStateWithFallback(cwd, sessionId).catch(() => null);
    if (!result?.ok) return null;
    const activeGoal = this.getGoalModeState()?.goal;
    const content = buildPabcdStageContent(result.value, activeGoal ? { objective: activeGoal.objective } : null);
    if (!content) return null;
    return { role: "custom", customType: "pabcd-stage-context", content, display: false, ... };
}
```
This message is pushed into the turn's message list at `agent-session.ts:5068` (`const pabcdStageMessage
= await this.#buildPabcdStageMessage(); if (pabcdStageMessage) messages.push(pabcdStageMessage);`),
right after the plan-mode and goal-mode messages and before user `prependMessages`.

The header content is a pure function over the lenient pabcd envelope:
`packages/coding-agent/src/session/pabcd-stage-header.ts:50-77` — `buildPabcdStageContent()` emits
`[PABCD — P: PLANNING]` + a `NEXT_HINTS[stage]` one-liner (e.g. "Present the plan draft and wait for
approval. When approved, run `jwc orchestrate a`."). It returns `null` for inactive/complete/unknown
stages, so the rail is fail-closed.

### 1.2 Session-scoped read (no cross-session leak)

`readPabcdStateWithFallback(cwd, sessionId)` at
`packages/coding-agent/src/jwc-runtime/orchestrate-state.ts:385-394`:
```ts
export async function readPabcdStateWithFallback(cwd, sessionId?) {
    if (sessionId) return await readPabcdState(cwd, sessionId);   // scoped-only when sessionId present
    return await readPabcdState(cwd);                             // shared fallback for standalone CLI
}
```
With a live `sessionId`, only `<cwd>/.jwc/state/sessions/<encoded-session-id>/pabcd-state.json` is read
— the legacy unscoped file is deliberately NOT consulted for live sessions (it leaked an I-stage header
into unrelated first turns). This mirrors codexclaw's `.codexclaw/sessions/<session>.json` scoped read
(`structure/22_session_storage.md` §Workflow state reads).

### 1.3 Self-transition (M3) — the model advances the FSM, not the runtime

jwc does NOT auto-advance the FSM on Stop. The model advances phases by running the exact shell
command at the end of each stage prompt:
`packages/coding-agent/src/prompts/jaw/orchestrate-p.md` (final line):
> ⛔ When the user approves the plan, run `jwc orchestrate a` yourself via the shell tool.

Same pattern at `orchestrate-a.md:19`, `orchestrate-b.md:16`, `orchestrate-c.md:17`,
`orchestrate-d.md:26`. After the CLI writes the new phase, `sendPabcdStageContext()` is called
(`agent-session.ts:4806`) to inject the new header immediately in the same turn (an in-process
"steer"). This is cli-jaw layer 2 (self-transition) + the immediate post-transition injection.

### 1.4 The documented 5-layer cli-jaw injection model

`structure/20_prompt_flow.md` §cli-jaw 주입 구조 대조 documents the 4-5 layers codexclaw must emulate:

| # | Layer | cli-jaw | jwc rail (99.03) |
|---|-------|---------|------------------|
| 1 | Resident guidance | `orchestration.md` section in system prompt (path pointers, "read before phase") | M1 `system-prompt.md` `<native-workflow orchestrate>` |
| 2 | Self-transition | "YOU advance phases by running `cli-jaw orchestrate …`" (model runs Bash) | M3 stage-prompt tail shell command |
| 3 | Per-turn header | `getPrefix()` — `[PLANNING MODE — User Feedback]` prefix every turn | M2 rail #10 `pabcd-stage-context` |
| 4 | Plan re-inject (compaction-immune) | A/B/C prepend full plan from DB every turn | M2b `plan_ref` prepend (file-sourced, regen each turn) |
| 5 | Post-compact handoff | `<overall_goal>`/`<current_state>` bootstrap | upstream compact summary + M2 per-turn regen covers it |

**Key takeaway for codexclaw:** layer 3 (per-turn header) is the compaction-immune mechanism. It works
because the header is REGENERATED FROM FILE every turn — it does not depend on conversation history.
Layer 2 (self-transition) is what changes the phase; layer 3 is what makes the new phase visible next
turn without the user re-typing.

## 2. The UserPromptSubmit-only constraint vs jwc in-process injection

### 2.1 jwc has an in-process server; codexclaw does not

jwc's `AgentSession` is a long-lived in-process object. It can call `#buildPabcdStageMessage()` and
`sendPabcdStageContext()` at any point during a turn — including immediately after the model runs
`jwc orchestrate a` (a tool call handled in-process). That immediate post-transition "steer" has NO
codex-plugin equivalent.

A Codex plugin hook fires ONLY at turn boundaries:
- `UserPromptSubmit` — fires when the user submits a prompt (start of turn).
- `Stop` — fires when the agent stops (end of turn).

There is no hook that fires mid-turn after a tool call. So codexclaw CANNOT emulate jwc's
`sendPabcdStageContext()` immediate-injection. It can only:
1. Inject at the START of the next turn (UserPromptSubmit), OR
2. Force a continuation at Stop (decision:block + reason → continuation fragment).

### 2.2 codex-rs hook output shapes (verified, hard constraint)

`codex-rs/hooks/src/schema.rs:433-441` — `UserPromptSubmitHookSpecificOutputWire`:
```rust
pub(crate) struct UserPromptSubmitHookSpecificOutputWire {
    pub hook_event_name: HookEventNameWire,
    #[serde(default)] pub additional_context: Option<String>,   // ← injection channel
}
```
`UserPromptSubmit` is the ONLY hook that can inject `additionalContext` (context prepended to the
turn). Verified in `codex-rs/hooks/src/events/user_prompt_submit.rs:parse_completed` —
`parsed.additional_context` → `additional_contexts_for_model`.

`codex-rs/hooks/src/schema.rs:439-448` — `StopCommandOutputWire` has NO `additional_context` field.
`Stop` can only: (a) do nothing, (b) `decision:"block"` + `reason` → a continuation fragment that
auto-continues the turn (`stop.rs:parse_completed` + `aggregate_results`). There is NO
`additionalContext` channel on Stop.

**Conclusion:** phase-directive injection MUST happen on `UserPromptSubmit`. Stop can only write state
or force a continuation. This rules out any design where Stop injects the directive text directly.

### 2.3 omo parity (output envelope)

omo `devlog/.lazycodex/plugins/omo/components/rules/src/hook-output.ts:10-16` confirms the exact stdout
shape codexclaw already uses:
```ts
{ hookSpecificOutput: { hookEventName, additionalContext } }
```
codexclaw `hook.ts:buildContextOutput` already matches this (CRLF normalize, trim, 32k cap). No change
needed to the output envelope.

## 3. Option table

| | (a) Stop advances FSM + pending marker → next UPS reads marker | (b) UPS always reads phase, injects header on phase-diff marker | (c) Hybrid: keep Pass-2 explicit trigger + add phase-diff auto-inject of short header |
|---|---|---|---|
| Injection channel | UPS (marker read) — correct channel | UPS — correct channel | UPS — correct channel |
| Who advances FSM | Stop hook (end of turn) | external (CLI/self-transition) | external (CLI/self-transition), Stop passive |
| Compaction-immune? | Partial — marker is a state field, but if marker is consumed once it's gone; relies on next UPS | Yes if header re-injected every turn; NO if diff-gated only (diff consumed → header lost post-compact within phase) | Yes — short header every turn (jwc M2 parity) + full directive re-injected on diff (file-sourced) |
| Ledger-spam risk | Stop advances every turn → must gate hard (canEnter open for A/C) | None — hook never writes ledger | None — hook never writes ledger |
| Stop complexity | High — Stop must run FSM gate + write phase + marker + avoid loop | None — Stop stays passive | None — Stop stays passive |
| jwc layer 2 parity | Breaks layer 2 (model no longer self-transitions; runtime auto-advances) | Preserves layer 2 (external CLI advances) | Preserves layer 2 (external CLI advances) |
| jwc layer 3 parity | Marker is consumed once → not true per-turn header | Diff-gated → not true every-turn header (compaction gap) | Every-turn short header = true layer 3 parity |
| Pass-2 compat | Breaks Pass-2 (Stop stops being passive) | Replaces Pass-2 trigger-only behavior | Additive — keeps Pass-2 explicit trigger intact |
| Failure mode | Stop FSM advance loops / false advances (A/C gates open) | Misses injection if diff-gate consumes + compact happens | Strict superset of Pass-2; lowest regression risk |

## 4. Recommendation: (c) Hybrid

**Recommend (c).** It preserves Pass-2 explicit-trigger behavior (zero regression), adds jwc-M2-parity
per-turn compaction-immune re-injection, keeps Stop passive (no FSM-advance loops, no ledger spam),
and respects the UserPromptSubmit-only injection constraint. Option (a) is rejected because it breaks
jwc layer 2 (self-transition) and risks Stop-ledger loops (the A/C gates in `fsm.ts:canEnter` are
unconditionally open, so a naive Stop advance would log every turn — the exact spam 018.2 deferred).
Option (b) is rejected because a pure diff-gate sacrifices compaction-immunity within a phase (once
the diff is consumed, a post-compaction turn in the same phase gets no header).

### 4.1 Design: three injection modes in handleUserPromptSubmit

On every `UserPromptSubmit`, after the same-turn idempotency check (`turn_id in injectedTurns`):

1. **Explicit trigger** (Pass-2 preserved): `detectTrigger(prompt)` matches → inject the FULL
   `phaseDirective(trigger)`, set `lastInjectedPhase = trigger`. The user explicitly asked; honor it.
2. **Phase-diff auto-inject** (NEW, the state-transition injection): no explicit trigger, but
   `state.phase !== state.lastInjectedPhase` → inject the FULL `phaseDirective(state.phase)` once,
   set `lastInjectedPhase = state.phase`. This is the "next turn after P→A carries the A directive
   automatically" behavior — no re-typed trigger needed.
3. **Per-turn short header** (NEW, jwc M2 compaction-immune): no explicit trigger, and
   `state.phase === state.lastInjectedPhase` → inject a SHORT 1-line stage banner
   `buildStageHeader(state.phase)` (jwc `pabcd-stage-header.ts` parity). Repeated every turn from
   file → compaction-immune (layer 3/4 parity).

Mode 2 fires once per phase entry (the transition turn). Mode 3 fires every subsequent turn in the
same phase. Together: the transition is announced once with the full directive, then reminded every
turn with a short banner that survives compaction.

### 4.2 New state field: `lastInjectedPhase`

One new field, added to the State interface, `defaultState`, AND the `readState` strict-reconstruct
literal (all three — the omo-style discipline requires any new field appear in all three or
`readState` drops it):

`state.ts`:
```ts
export interface State {
  phase: Phase;
  sessionId: string;
  slug: string;
  updatedAt: string;
  flags: Flags;
  supersededBy: string | null;
  injectedTurns: string[];
  lastInjectedPhase: Phase | null;   // NEW — phase whose full directive was last injected
}

export function defaultState(sessionId: string, slug = ""): State {
  return {
    phase: "I",
    sessionId,
    slug,
    updatedAt: new Date().toISOString(),
    flags: { interview: false, auditPassed: false, checkPassed: false },
    supersededBy: null,
    injectedTurns: [],
    lastInjectedPhase: null,          // NEW
  };
}

// inside readState strict-reconstruct literal (spread-less, additive line):
    lastInjectedPhase:
      typeof parsed.lastInjectedPhase === "string" && PHASES.includes(parsed.lastInjectedPhase as Phase)
        ? (parsed.lastInjectedPhase as Phase)
        : null,                       // NEW — reject unknown phases, default null
```
No other state fields change. `Flags`, `LedgerEntry`, `writeState`, `appendLedger` untouched.

### 4.3 hook.ts changes (diff-level sketch)

Add the short header builder (jwc `pabcd-stage-header.ts:STAGE_LABELS` parity):
```ts
const STAGE_LABELS: Record<Phase, string> = {
  I: "INTERVIEW", P: "PLAN", A: "AUDIT", B: "BUILD", C: "CHECK", D: "DONE",
};
export function buildStageHeader(phase: Phase): string {
  return `[codexclaw — ${phase}: ${STAGE_LABELS[phase]}]`;
}
```

Rewrite `handleUserPromptSubmit` (replaces the current trigger-only body):
```ts
export function handleUserPromptSubmit(payload: UserPromptSubmitPayload): string {
  if (payload.hook_event_name !== "UserPromptSubmit") return "";
  const turn = payload.turn_id ?? "";
  const state = readState(payload.cwd, payload.session_id);
  if (turn && state.injectedTurns.includes(turn)) return "";   // same-turn idempotency (Pass-2)

  const trigger = detectTrigger(payload.prompt);
  let directive: string;
  let nextLastInjected: Phase;

  if (trigger) {                                        // mode 1 — explicit (Pass-2 preserved)
    directive = trigger === "I" ? interviewDirective() : phaseDirective(trigger);
    nextLastInjected = trigger;
  } else if (state.phase !== state.lastInjectedPhase) { // mode 2 — phase-diff auto-inject (NEW)
    directive = phaseDirective(state.phase);
    nextLastInjected = state.phase;
  } else {                                              // mode 3 — per-turn short header (NEW)
    directive = buildStageHeader(state.phase);
    nextLastInjected = state.lastInjectedPhase;         // unchanged
  }

  if (turn) {
    writeState(payload.cwd, {
      ...state,
      injectedTurns: [...state.injectedTurns, turn],
      lastInjectedPhase: nextLastInjected,
    });
  }
  return buildContextOutput("UserPromptSubmit", directive);
}
```
`handleStop` stays passive (returns `""`, writes nothing) — unchanged from Pass-2. The phase advance
itself is performed by the self-transition CLI (`codexclaw orchestrate <phase>`, jwc layer 2 parity,
a separate pass), which writes `state.phase` + the ledger entry. The hook only READS state and
injects; it never advances the FSM and never calls `appendLedger`.

### 4.4 Idempotency key

Two-level, both file-sourced (compaction-immune — they live in `.codexclaw/sessions/<session>.json`,
not conversation history):
- **Same-turn:** `injectedTurns` contains `turn_id` → suppress (handles hook re-fire in one turn).
- **Cross-turn, per-phase:** `lastInjectedPhase === state.phase` → downgrade mode 2 (full) to mode 3
  (short header). The full directive auto-injects ONCE per phase entry; subsequent turns get only the
  short banner.

### 4.5 Ledger-spam avoidance

The injection hook NEVER calls `appendLedger`. Ledger entries are written ONLY by the self-transition
CLI on an actual phase change (one line per transition). `lastInjectedPhase` and `injectedTurns` are
state-file fields, not ledger rows. This is the direct fix for the 018.2 concern ("a naive advance
would log every turn") — the hook doesn't advance, so it can't spam the ledger.

### 4.6 Compaction-immunity (jwc layers 4/5)

- Layer 3 (per-turn header): mode 3 re-injects the short banner every turn from the state file.
  Compaction clears conversation history but not the file; the next `UserPromptSubmit` re-reads
  `state.phase` and re-injects. This is exactly jwc `#buildPabcdStageMessage` regen-from-file.
- Layer 4 (plan re-inject): not in this doc's scope (the `plan_ref` prepend is a later pass), but the
  same file-sourced-every-turn principle applies when added.
- Layer 5 (post-compact handoff): codex upstream compaction summary + mode 3 per-turn regen covers
  this — no separate bootstrap needed, mirroring jwc's decision (`20_prompt_flow.md`: "upstream
  컴팩션 요약 + M2 매 턴 재생성으로 갈음").
- The diff detection (mode 2) is also compaction-safe: `lastInjectedPhase` is in the state FILE, so
  even after compaction the comparison `state.phase !== state.lastInjectedPhase` runs against durable
  file state, not lost conversation state.

### 6.3 Which pass this lands in

Per the 7-pass map in `017_pabcd_loop_plan.md`:

| Pass | Unit | Status |
|------|------|--------|
| 1 | IPABCD state engine (state.ts, fsm.ts) | ✅ done |
| 2 | Directive hook (UserPromptSubmit trigger inject + passive Stop) | ✅ done (018.2) |
| **3** | **Goal budget gate (T-023 PreToolUse ^create_goal$)** | **← this design's changes land here** |

This design is an **incremental hardening of the Pass-2 hook**, not a new pass unit. It should be
implemented as the first task of **Pass 3** (or a 018.2.1 point release) because:
- Pass 3 has no upstream dependency on Pass 2's Stop behavior (per 017 notes, Pass 3 is
  parallel-ok), so it can absorb the hook rewrite without blocking the goal-gate work.
- The self-transition CLI (jwc layer 2, `codexclaw orchestrate <phase>`) that WRITES `state.phase`
  and the ledger is a prerequisite for mode 2 to observe a diff. If the CLI lands in Pass 3, the
  hook hardening should land alongside or immediately after it. If the CLI is deferred further, mode
  2 still works the first time (diff from `null` → `state.phase`), and the explicit trigger (mode 1)
  remains the manual advance path until the CLI exists.
- Passes 4-7 (dev-* skills, roles, install, build) are unaffected; they consume the injected
  directive text unchanged.

**Dependency note:** the full state-transition-time guarantee ("P→A, next turn carries A directive
automatically") requires BOTH (i) the self-transition CLI writing `state.phase` (layer 2) AND (ii)
this hook's mode-2 diff detection (layer 3). This doc designs (ii) and assumes (i) is the
layer-2 advance mechanism. If (i) is not yet built, mode 2 still fires on the first turn of any
session (diff from `null`) and on any externally-written phase change.
