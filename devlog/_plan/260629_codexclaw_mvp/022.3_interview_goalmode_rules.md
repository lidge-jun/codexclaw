# 022.3 — Interview ⇄ Goal-Mode Rules (HARD)

Status: DESIGN (source-verified)  ·  Phase 1
Decisions (jun):
- Interview HARD-depends on `request_user_input` (feature `default_mode_request_user_input`).
- In GOAL MODE, interview / user-input prompting is ABSOLUTELY FORBIDDEN.

## Source ground truth
- Tool contract: `protocol/src/request_user_input.rs`:
  - `RequestUserInputArgs { questions: Vec<RequestUserInputQuestion> }`
  - `RequestUserInputQuestion { id, header, question, isOther, isSecret, options?: [{label, description}] }`
  - Response: `RequestUserInputResponse { answers: HashMap<id, { answers: Vec<String> }> }`
- codex already couples goal continuation with user activity:
  - `core/src/goals.rs`: "user activity suppresses the next automatic continuation…"
  - test `pending_..._does_not_spawn_extra_goal_continuation` — pending user-input must NOT spawn a
    goal continuation. So goal mode + request_user_input interaction is a known codex concern.

## Rule: HARD dependency for Interview
- Interview phase uses `request_user_input` directly (no plain-text fallback). Install MUST enable
  `default_mode_request_user_input` (028.1). If unavailable → Interview is unavailable (fail-fast,
  report), NOT silently degraded.

## Rule: FORBID interview/user-input in Goal Mode (NON-NEGOTIABLE)
- When a codex goal is active (goal mode = autonomous self-advance), codexclaw MUST NOT trigger the
  Interview phase nor call `request_user_input`.
- Enforcement (defense in depth):
  1. UserPromptSubmit hook: if goal active, do NOT inject interview directive; suppress I-phase.
  2. PreToolUse hook matcher on `request_user_input`: if goal active → `permissionDecision: "deny"`
     with reason "user-input/interview is forbidden in goal mode; proceed autonomously".
  3. `ipabcd` skill text: explicit "never interview in goal mode" rule.
- Rationale: goal mode is self-advancing with full authority; blocking on user input breaks the
  autonomous loop (matches cli-jaw goal-continuation philosophy + codex goals.rs suppression).

## Detecting goal mode
- Q-GM-1 (verify in build): how a hook learns a goal is active — env var, payload field, or a
  goal-state file. Check `core/src/goals.rs` + hook payload schema before implementing the deny.

## Verify
- Interview works when no goal active + flag on.
- With a goal active: I-phase never triggers; `request_user_input` call is denied with the reason.
- Pending interview never spawns a goal continuation (mirror codex test).
