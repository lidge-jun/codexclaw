## FINDING A (HIGH) — goal-active is NOT in hook payload  [blocks 022.3 interview-deny]
- Neither UserPromptSubmit nor PreToolUse payload carries a "goal active" field.
- omo does NOT detect goal-active at all — its create_goal gate is STATELESS (only inspects whether
  the create_goal call carries token_budget). So omo gives no reusable recipe for "forbid X while a
  goal is active".
- Impact: "forbid interview/request_user_input in goal mode" (022.3 rule 2) cannot be implemented by
  reading the hook payload alone.
- Options to resolve (pick in next round):
  - A1: Read goal state from thread store / a goal-state file via `session_id`+`transcript_path`
        in the hook (needs the store path/format — extra research, possible instability).
  - A2: Rely on codex's OWN goal-mode suppression (goals.rs already suppresses continuation on user
        activity / pending input) and make codexclaw's rule ADVISORY in the `ipabcd` skill text
        ("never interview in goal mode") instead of a hard hook deny.
  - A3: Hybrid — skill-advisory now (Phase 1), hard hook deny later once the goal-state read path is
        proven (defer the deny to a follow-up).
- NOTE: jun's instruction is HARD "absolutely forbid in goal mode". A1 is the only TRUE hard-enforce
  path; A2/A3 are softer. Decision required (Q-GM-1).
