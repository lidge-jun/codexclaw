## SKILL.md frontmatter — EXACT accepted fields
From `SkillFrontmatter` (loader.rs:40):
```yaml
---
name: <string>                      # optional
description: <string>               # optional — THIS drives routing
metadata:
  short-description: <string>       # optional (note the HYPHEN, serde rename)
---
```
That is ALL the frontmatter struct accepts. There is NO `keywords`, NO `search_terms`,
NO `default_prompt` in frontmatter. (cli-jaw's `metadata: { "keywords": [...] }` is NOT read.)

## Separate metadata file: `openai.yaml`
Constant `SKILLS_METADATA_FILENAME = "openai.yaml"` (loader.rs:110). Lives in the skill dir.
Shape (`SkillMetadataFile` → `Interface`/`Dependencies`/`Policy`):
```yaml
interface:
  display_name: <string>
  short_description: <string>
  icon_small: <path>
  icon_large: <path>
  brand_color: <string>
  default_prompt: <string>
dependencies:
  tools:
    - type: <string>          # e.g. mcp
      value: <string>
      description: <string>
      transport: <string>
      command: <string>
      url: <string>
policy:
  allow_implicit_invocation: <bool>   # KEY: enables auto-routing without explicit $skill
  products: [ ... ]
```
NOTE: omo ships `skills/<skill>/agents/openai.yaml` with `search_terms` — codex-rs `Interface`
does NOT define `search_terms`, so it is ignored by this loader version. Use `description`
(frontmatter) for routing text, and `openai.yaml interface` for display/default_prompt.

## Routing model (critical for conversion)
- codex auto-selects skills primarily via `description`. So conversion MUST rewrite each
  cli-jaw `description` into a trigger-rich line (omo "MUST USE for ..." style).
- `policy.allow_implicit_invocation: true` lets a skill activate without the user typing `$name`.
  Decide per skill whether implicit invocation is desired (e.g. `dev` always-on vs `dev-uiux`
  only-when-relevant).

## Skill discovery roots (loader.rs)
- `~/.agents/skills` (AGENTS_DIR + SKILLS_DIR) — home scope.
- Project scope via config layers; marker-based.
- Plugin-provided skills carry `plugin_id` / `plugin_root` (so our plugin's `skills/` is picked up).
- Files starting with `.` are skipped; symlinks handled explicitly.

## config side (skills_config.rs)
- `SkillsConfig.include_instructions` toggles the automatic skills-instructions block per turn.
- `SkillsConfig.config[]` = per-skill `{ path|name, enabled }` selectors.
- `bundled.enabled` controls bundled skills.
- `deny_unknown_fields` everywhere → unknown keys ERROR. Conversion must not emit stray fields.

## Hooks
- Hook events used by omo and accepted by codex: SessionStart, UserPromptSubmit, PreToolUse,
  PostToolUse, PostCompact, Stop, SubagentStop.
- Hook JSON shape (verified from omo): `{ "hooks": { "<Event>": [ { "hooks": [ { "type":
  "command", "command": "...", "timeout": N, "statusMessage": "..." } ], "matcher": "^create_goal$" } ] } }`.
- PreToolUse can return `permissionDecision: "deny"` + `permissionDecisionReason` to block a tool.

## Implications for codexclaw conversion (feeds 024)
1. Frontmatter: keep ONLY name/description/metadata.short-description. Drop cli-jaw `keywords`.
2. Put display/default_prompt into `openai.yaml interface` if needed (optional).
3. Set `policy.allow_implicit_invocation` deliberately per skill.
4. `dependencies.tools` is where a skill can declare an MCP/tool need (e.g. future ocx bridge).
5. `deny_unknown_fields` → validate every converted skill against this exact schema.
