## Design decision: compile with node:module.stripTypeScriptTypes, no external toolchain
`scripts/build.mjs` (real) will, for each component under `components/*` that has `src/`:
1. Read every `src/**/*.ts`.
2. `stripTypeScriptTypes(source, { mode: "strip" })` to get JS (A-audit blocker 1: the only valid modes are
   "strip" | "transform"; "strip-only" throws `ERR_INVALID_ARG_VALUE` — empirically verified on all 10
   component sources). "strip" keeps line layout; no downleveling needed on Node 24. NOTE: the API is
   ExperimentalWarning on v24 — the build/test must tolerate that stderr line (don't assert empty stderr).
3. Rewrite relative import/export specifiers ending in `.ts` → `.js` (regex on
   `from "..."` / `import("...")` with a leading `./` or `../`).
4. Write to `components/<c>/dist/<same-relative-path>.js`.
5. Skip `test/` (tests run on `.ts` directly via `node --test`).
Rationale: matches the existing manifest references (dist/*.js), needs zero new dependencies, and is
fully reproducible (same input → same output). A `tsc`/`esbuild` path would add a toolchain the repo
deliberately avoids (components have no build step today and run `.ts` natively). NO-BUNDLER CAVEAT
(A-audit): strip-without-bundle is sound ONLY because all 4 components have zero third-party runtime
deps (every package.json `dependencies` is `{}`) and import only `node:*` + relative `./x.ts`. If any
component later adds a third-party import, a bundler (omo uses `bun build`) would be required so the
bare specifier resolves at the shipped dist path. Specifier rewrite is LINE-ORIENTED post-strip (the two
multi-line `import {...} from "./features.ts"` blocks in config-guard land their `from "..."` on its own
line after strip, so a per-line regex still catches them; `node:`/package specifiers never end in `.ts`).
