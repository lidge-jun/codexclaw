PABCD Emergence Gap

왜 같은 루프가 빌드/버그엔 강하고 알고리즘 같은 창발 태스크엔 약한가 — NYPC 봇 3.5/8 정체의 구조적 원인

핵심 한 줄: PABCD는 수렴(exploitation) 기계지, 발산(divergence) 기계가 아니다. 모든 페이즈가 이미 정해진 한 타겟으로 좁혀 들어간다. 타겟이 주어진 빌드/버그엔 그게 곧 정답이지만, 타겟을 찾아야 하는 알고리즘 태스크엔 첫 접근법에 갇혀 증상만 패치하게 된다.

01 수렴 vs 발산 — 빠진 축

있음 · PABCD가 하는 것

수렴 (Exploitation)

  • 한 접근법을 깊게 파고든다
  • 버그를 잡고 정확성을 높인다
  • 증거의 모양을 검증한다 (exit 0 / PASS)
  • 페이즈를 순서대로 전진시킨다
없음 · 창발 태스크에 필요한 것

발산 (Divergence)

  • 경쟁하는 접근법 N개를 만든다
  • 진짜 목표 점수로 서로 경쟁시킨다
  • 다양성 아카이브를 유지한다
  • 정체 시 탐색을 주입한다

03 두 개의 구조적 뿌리 (file:line)

① 목표-델타 기억이 없다. state.ts:18은 phase/flag/Stop카운터만 저장. ledger(state.ts:36)는 전이+증거 문자열만, 메트릭 델타는 없음 → 루프가 정체를 알 수가 없다.

② C→D가 목표가 아니라 명령 모양에 묶여 있다. attest.ts:98은 checkOutput+exitCode만 게이트 → 공식 저지인지 약한 자가대전인지 구분 못 함. 로컬 96%가 C/D를 영원히 통과하는데 진짜 점수는 평평.

06 고칠 방향 (정직한 E-tier)

PABCD를 갈아엎지 않는다 — 앞에 발산 패스를 붙인다: Diverge → (후보마다 PABCD) → Select → Archive → loop

레버메커니즘강제 가능 tier한계

레버 2는 jawcode autoresearch를 차용, 가짜완료 방지는 lazycodex SubagentStop을 차용. 레버 1·3·4(발산 레이어)는 셋 다 없어 codexclaw가 처음 만드는 차별점. 원격 점수 자동 검증은 불가능(훅이 못 봄) — 운영자 입력으로 정직하게 둔다.

08 이건 loop 개선이기도 하다 (Ralph · autoresearch · long-running harness)

cxc-loop은 지금 "한 work-phase = 한 PABCD 사이클" 계약뿐 — Ralph의 디스크에 박힌 plan을 매 패스 다시 읽기도, autoresearch의 baseline/best 메트릭 기억도 없다. 정체 캡이 점수가 아니라 턴 수를 센다. 그래서 발산 작업은 loop 개선으로도 박아야 한다.

패턴 (출처)cxc-loop 개선tier
L1 디스크 plan 재독 Ralph·harness매 패스 .codexclaw/plan/* 재독, 진행은 디스크에 (fresh context 부활)E7+E2
L2 메트릭 기억 autoresearch·SICAwork-phase마다 true-objective 메트릭 + baseline/best를 ledger에 carryE2/CLI
L3 정체→발산 autoresearch·noveltyturn-cap을 메트릭-델타 체크로: N패스 정체 시 Stop이 "diverge" 주입E2 (핵심)
L4 아이디어 백로그+아카이브 autoresearchideas backlog + kept-candidate 아카이브를 .codexclaw/에 영속E2
L5 keep/discard+confidence autoresearch패스 기록을 메트릭 keep/discard로 (discard 시 워크트리 revert)E7+E2
L6 fresh-context 부활 Ralph·harness압축 후 plan+메트릭+아카이브를 메모리 아닌 디스크에서 reloadE2

Ralph = "fresh context + 디스크 상태 + while 루프". codexclaw의 Stop-hook 연속은 그보다 강하지만(종료를 게이트) 새 접근을 작성하진 못한다 — "more of the same"를 거부할 뿐. 진짜 런타임 레버는 L3 하나(Stop의 메트릭-델타 정체 체크), 나머지는 loop가 재독하는 파일을 에이전트가 유지하는 구조. 전부 no-server.

09 언제 발산하나 + 어떻게 고르나 (gating + selection — 핵심 결정)

발산은 기본값이 아니라 예외다. 대부분의 목표 빌드는 한 전략으로 충분하다. 판별선은 태스크 종류가 아니라 목표의 성격:

충족형 → 단일 전략 (N=1)

Satisfy-spec

  • pass/fail, 정답 동작 하나
  • 로컬 검증 가능 (테스트가 settle)
  • 지형 not deceptive
  • 버그픽스 · CRUD · 리팩터 · 마이그레이션
최대화형 → 발산 (N≥2)

Maximize-metric

  • 연속 점수, "더 나음"이 열려있음
  • 로컬 검증 어려움 (원격 저지/holdout)
  • 지형 deceptive — 로컬 프록시가 속임
  • NYPC 봇 · perf 튜닝 · 휴리스틱 탐색

핵심: 발산은 보통 선제가 아니라 정체-트리거다. 항상 N=1로 싸게 시작하고, 진짜 메트릭이 정체했을 때만 발산 모드로 들어갔다 빠져나온다:
